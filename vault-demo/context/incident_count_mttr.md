# Incident Count / MTTR

**Owner:** Tech · **Источник данных:** Monitoring

## Что это и зачем

Количество P1/P2 инцидентов и MTTR (Mean Time To Restore, среднее время восстановления). 1 час даунтайма в пик = потеря 3–5% дневного GGR. Это прямая связь между скоростью реакции и деньгами.

## Формула — детали

Числитель: Count P1/P2 incidents; Avg(restore − start)

Знаменатель: Per month

## Ловушки

1) Целевые MTTR: P1 <15 минут, P2 <1 час, P3 <8 часов. 2) «Инфляция» P1, когда всё подряд помечают как P1, severity теряет смысл. 3) Количество инцидентов без трендового анализа бесполезно, важна динамика.

## Как улучшать

1) Приоритеты: платёжный шлюз (P1 при любом даунтайме), game API (P1 при >5 мин), лобби (P1 при >10 мин). 2) PSP failover менее 30 секунд, автоматическое переключение на backup PSP. Таблица маршрутизации с 2-3 PSP на метод оплаты, экономит до 95% потенциальных потерь. 3) Observability: дашборд с deposit success rate, game launch rate, bet rate, видим проблему до того, как доложат. 4) Blameless post-mortem в течение 48 часов.

## Алерты (исходный текст)

Red: >5 P1/mo

Yellow: >3 P1/mo
