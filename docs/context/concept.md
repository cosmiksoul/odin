# 00 · Концепция ODIN

## Откуда имя

**Один** — главный скандинавский бог, всеведущий, стремился познать всю магию мира. Его восприятие дополняли два ворона:

- **Хугин** (др.сканд. *Huginn*) — "мысль". Летал над миром и рассказывал Одину о происходящем.
- **Мунин** (др.сканд. *Muninn*) — "память". Хранил знание обо всём, что уже было.

Метафора проекта:

- **ODIN** (фронт) — сам бог. Это пользовательская поверхность, с которой работает человек. Здесь он видит, знает, принимает решения.
- **Хугин** (MCP) — ворон мысли. LLM-агент, который "летает над данными" и формирует гипотезы.
- **Мунин** (semantic layer + DWH) — ворон памяти. Хранит **две половины памяти**: значения метрик за всё прошедшее время (DWH / Parquet) **и** смысл этих метрик — описания, ловушки, связи, changelog, залогированные аномалии (MD-файлы в git). См. decisions-log.md ADR-007.

Два ворона служат богу. В архитектуре — два бэкенд-сервиса служат фронту.

## Упрощение концепции

Изначальная идея была шире — с внешними источниками: regulatory, stocks, prediction markets, competitor pulse. В процессе обсуждения упростили до безобразия:

```
Фронт (каталог + UI)  =  ODIN
DWH (живые данные)    =  Мунин
MCP (LLM-интерфейс)   =  Хугин
```

Всё остальное — **out of scope для этого проекта**. Старая идея iGaming Intelligence MCP остаётся в 00-Ideas как потенциально отдельный проект.

                          │         ОДИН         │
                          │   (фронт / каталог)   │
                          │ decision OS for iGaming │
                          └──────────┬───────────┘
                                     │
                          использует ↓↑ сервисы
                                     │
                 ┌───────────────────┴───────────────────┐
                 │                                       │
           ┌─────▼──────┐                         ┌──────▼─────┐
           │   МУНИН    │                         │   ХУГИН    │
           │   память   │                         │   мысль    │
           │            │                         │            │
           │  MD в git  │◄───────запросы──────────│    MCP     │
           │     +      │                         │  сервер    │

Один — это и проект целиком, и конкретно его фронтенд. Два ворона (Хугин и Мунин) — бэкенд-сервисы, которые обслуживают Одина.

## Функциональные связки

- **ODIN → Мунин:** UI читает MD-спеки (рендер карточки) и значения из DuckDB (текущее, тренд, WoW)
- **Мунин → ODIN:** значения сопоставляются с порогами из спеки → алерт срабатывает (red/yellow/ok)
- **Хугин → Мунин (spec):** LLM получает MD-спеку метрики — формулу, ловушки, зависимости, **прошлые аномалии**
- **Хугин → Мунин (values):** LLM получает актуальные и исторические значения по разрезам через DuckDB-запросы
- **Хугин = синтез:** "метрика просела, почему?" → LLM видит спеку (ловушки), данные (тренды), историю (прошлые инциденты), формирует гипотезу
- **Data-team компании → Мунин (sync):** компания готовит Parquet-витрину в стандартизированной схеме, кладёт в указанное место, ODIN автоматически синхронизирует. См. munin-sync-contract.md.
- **Автор/аналитик → Мунин (write spec):** правит метрики **прямо в ODIN** через встроенный WYSIWYG-редактор. Под капотом сохраняется в MD-файл + локальный git commit. История изменений видна в карточке метрики. Никакого внешнего git/CLI/MD-редактора знать не нужно — UX как у Notion. См. decisions-log.md ADR-011.

## Главное отличие от классических BI

В Tableau/Looker ты видишь число, но не знаешь, что оно значит и где ловушки. В ODIN — карточка метрики показывает **и значение, и определение, и пороги, и контекст, и связи с другими метриками**.

## UVP (working)

> **Privacy-first BI для регулируемых индустрий с локальной LLM.** Десктопное приложение. Данные не покидают вашу машину. Модель — любая, какую поставите. Семантический слой команды — в одном месте. Root cause analysis с институциональной памятью.

**Сильные стороны:**

1. **Privacy-first архитектура.** Десктопное приложение, локальная БД, опциональная локальная LLM (Ollama / LM Studio). Data leakage невозможна — всё на машине пользователя. Критично для iGaming, финтеха, healthcare, крипто. См. decisions-log.md ADR-008.
2. **Push-based интеграция через Parquet-витрину.** ODIN не лезет в ДВХ компании. Data-team готовит одну витрину в стандартизированной схеме, ODIN её читает. Никакого ETL внутри ODIN, никаких коннекторов к Snowflake/BigQuery. Security-команды это любят. См. decisions-log.md ADR-010 и munin-sync-contract.md.
3. **Спека и данные в одном месте.** Не нужно переключаться между документом с описанием метрик и BI-дашбордом. Карточка метрики показывает определение, формулу, ловушки и текущие значения сразу.
4. **Institutional memory built-in.** Решает главную проблему BI: "знание о метрике умирает в головах команды". История аномалий, решённых инцидентов, проверенных гипотез — живёт в локальном git, доступна и людям, и LLM.
5. **Bring-your-own-model.** Поддержка Ollama, LM Studio, любого OpenAI-compatible endpoint, Claude Desktop через MCP. Параноикам — локальная Llama 3.1 / Qwen 14B. Тем, кому policy позволяет — Claude.
6. **MCP-ready архитектура.** Один data layer обслуживает и человека (UI), и LLM (MCP). LLM-агент видит то же, что видит аналитик в карточке.

**Чем отличаемся от Tableau / Looker / Hex / Mode:**

- Они — SaaS, требуют отправки данных в их облако (или сложного self-hosted с поддержкой)
- Они — pull-based, требуют коннекторов к ДВХ, credentials, network access
- У них нет институциональной памяти — спеки в одном месте (Confluence), данные в другом (BI)
- Они не работают с локальными LLM — все интеграции "with AI" идут через GPT/Claude API

ODIN — наоборот. Это **инструмент эпохи local-first AI**.

## Ключевой demo-сценарий (root cause analysis с локальной LLM)

```
0. Контекст для зрителя (закадровый текст в видео):
   "ODIN установлен у меня на ноутбуке. Подключён к локальной модели
   Qwen 14B через Ollama. Никакого cloud. Данные — Parquet-витрина,
   которую data-team нашей выдуманной компании положила в нашу
   общую папку. Метрики не покидают мой ноутбук."

1. Пользователь открывает ODIN, видит плитку GGR Margin → просадка 15% WoW

2. Кликает "Investigate with Хугин"

3. Чат с локальным агентом: "Разберись почему GGR Margin просел 18-22 апреля"

4. Агент через MCP:
   → читает спеку (MD): формула, ловушки, зависимости, разрезы
   → читает ИСТОРИЮ (anomaly-заметки): были ли похожие аномалии
   → читает данные (DuckDB): значения метрики и связанных по разрезам
   → формирует гипотезу: "Похожий паттерн был в марте 2026,
      причина была bonus abuse в Бразилии. Проверим те же разрезы..."

5. Пользователь: "Покажи bonus cost по Бразилии"

6. Агент → график/таблица в ответ

7. После расследования: пользователь (или агент) создаёт anomaly-заметку
   в локальном git → она остаётся в истории метрики навсегда

8. Финальный кадр видео:
   "Полная диагностика, данные не покинули мой ноутбук.
   Это ODIN."
```

Это то, что делает из ODIN не просто "красивый BI", а **root cause analysis ко-пилот с доменной экспертизой, накопленной институциональной памятью, и privacy-first архитектурой**.

Этот сценарий — **один из ключевых для demo-видео**, но не единственная цель MVP. MVP должен быть **целостным продуктом для аналитиков** — с редактируемыми спеками, history UI, локальной LLM. См. decisions-log.md ADR-011.
