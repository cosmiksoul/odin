# JBTD — Job To Be Done (MVP scope)

> Полный список user stories для MVP ODIN. Группировка по областям продукта. Теги милестоунов — для понимания "к какому слою относится фича", не для дисциплины спринтов (см. JTBD-full-scope.md про связь milestones и спринтов).

> **Статус:** валидирован автором 2026-04-25. Все ранее открытые вопросы закрыты в пользу включения в MVP. Финальный список переедет в `<project>/docs/JTBD-full-scope.md` под управление Cowork.

> **MVP разбит на два этапа** (см. ADR-016): **MVP-1** — read-only Catalog (M1 + M2 + опционально M3 + частично M4), **MVP-2** — full ODIN с WYSIWYG/sync/LLM (M-CRED, M-SYNC, M-EDIT, M5-M9). Теги `[M-N]` остаются как есть; разделение по MVP-1/MVP-2 — на уровне milestones, не stories.

## 3. Редактирование смысла метрики [M-EDIT] ★

> Критично для ICP — аналитики не используют git/CLI. См. decisions-log.md ADR-011.

### Базовое редактирование

- [ ] ★ ◆ Как пользователь я хочу **внести изменения в описание метрики через карточку метрики посредством WYSIWYG-редактора**, чтобы править спеки без знания markdown/git `[M-EDIT]`
- [ ] ★ Как пользователь я хочу **редактировать frontmatter-поля как формочки** (dropdown для уровня, input для формулы и т.д.), чтобы менять структурированные данные без работы с YAML `[M-EDIT]`
- [ ] ★ Как пользователь я хочу видеть **индикатор статуса сохранения** ("Сохранено", "Сохраняется..."), чтобы быть уверенным что мои правки не потеряются `[M-EDIT]`
- [ ] Как пользователь я хочу **отменить изменения** до сохранения (Esc / Cancel button), чтобы не бояться экспериментировать в редакторе `[M-EDIT]`

### CRUD метрик в каталоге

- [ ] Как пользователь я хочу **добавить новую метрику** через UI (создать с нуля, заполнить поля, сохранить), чтобы расширять каталог своими метриками `[M-EDIT]`
- [ ] Как пользователь я хочу **удалить метрику** из каталога с подтверждением, чтобы убирать ненужные `[M-EDIT]`

### Производные метрики (derived metric с фильтром)

> Гибридный подход с двусторонней связью anomaly-заметок. См. **архитектурное примечание** ниже.

- [ ] Как пользователь я хочу **создать дубликат метрики с зашитым фильтром** (например, "GGR · filtered by GEO:US" из родительской GGR), чтобы мониторить срез метрики как отдельную единицу в каталоге `[M-EDIT]`
- [ ] Как пользователь я хочу видеть **производные метрики в каталоге как отдельные карточки** с пометкой "filtered: GEO=US" и ссылкой на родительскую, чтобы понимать происхождение `[M-EDIT]`
- [ ] Как пользователь я хочу **писать anomaly-заметки в производной метрике**, чтобы фиксировать инциденты на конкретном срезе (например, "просадка GGR в US 18 апреля") `[M-EDIT, M6]`
- [ ] Как пользователь я хочу чтобы **anomaly-заметки производной метрики отражались в родительской** (двусторонняя связь), чтобы при просмотре GGR видеть и общие инциденты, и инциденты её срезов — *см. архитектурное примечание* `[M-EDIT, M6]`
- [ ] Как пользователь я хочу **редактировать производную метрику независимо от родительской** (свой текст, свои ловушки, свои бенчмарки), чтобы не быть привязанным к шаблону родителя `[M-EDIT]`

> **Архитектурное примечание (для M-EDIT и M6):**
> Гибридный подход означает что производная метрика — это **самостоятельная сущность** в каталоге (свой `metric_id`, своя карточка, свой набор anomaly-заметок), но **технически связана** с родительской через ссылку и фильтр. Anomaly-заметка производной метрики **видна и в карточке родительской** (как "инцидент в срезе"), но не наоборот — общие инциденты родительской не дублируются в каждый срез.
>
> Это требует:
> - Поля в frontmatter spec.md производной: `parent_metric_id`, `filter` (например, `geo: us`)
> - Поля в frontmatter anomaly.md: помимо `metric_id` — опциональный список `also_visible_in: [parent_metric_id]`
> - В UI карточки родителя: секция "Anomaly от срезов" отдельно от собственных
>
> **Это open question по реализации, не блокер MVP.** Финализация формата — в M-EDIT.

## 7b. Vault sync — shared semantic layer для команды [M-SYNC]

> Поэтапная реализация. См. decisions-log.md ADR-013. В MVP — single-user + опциональный git remote, без conflict UI и RBAC.

### Подключение и базовый sync

- [ ] Как пользователь я хочу **подключить мой vault к git remote** (GitHub / GitLab / Gitea / корпоративный bare git), чтобы синхронизировать метрики с командой `[M-SYNC]`
- [ ] Как пользователь я хочу **указать credentials к git remote** (SSH-ключ или Personal Access Token) через credentials store, чтобы push/pull работали без CLI `[M-SYNC, ADR-012]`
- [ ] Как пользователь я хочу **вручную инициировать push** через кнопку в Settings → Vault sync, чтобы отправить свои изменения коллегам `[M-SYNC]`
- [ ] Как пользователь я хочу **вручную инициировать pull** через кнопку, чтобы получить изменения коллег `[M-SYNC]`

### Автоматизация

- [ ] Как пользователь я хочу **auto-pull при старте ODIN** (опция, default off), чтобы при запуске видеть свежую версию vault'а `[M-SYNC]`
- [ ] Как пользователь я хочу **auto-push при save метрики** (опция, default off), чтобы изменения сразу попадали в shared vault — *по умолчанию off, чтобы избежать спама коммитов* `[M-SYNC]`
- [ ] Как пользователь я хочу видеть **timestamp последнего успешного pull/push** в Settings, чтобы понимать когда синхронизировался `[M-SYNC]`

### Понятная индикация конфликтов

- [ ] Как пользователь я хочу **видеть понятное сообщение при git-конфликте** ("Анна изменила GGR в 14:32, ваша версия не может быть запушена пока конфликт не разрешён"), чтобы понимать что произошло `[M-SYNC]`
- [ ] [post-MVP] Как пользователь я хочу **разрешать конфликты внутри ODIN** (не через git CLI), чтобы не выходить из приложения — *Этап 2 ADR-013, post-MVP* `[post-MVP]`

### Что НЕ в MVP (зафиксировано в ADR-013)

- ❌ Auto-write Хугина в shared vault — в MVP только "Save as anomaly note" по кнопке
- ❌ UI разрешения конфликтов внутри ODIN — fallback на git CLI с понятным сообщением
- ❌ Real-time sync (как Obsidian Sync) — только manual + auto-pull при старте
- ❌ RBAC (видимость метрик по ролям) — Этап 3 (post-MVP)
- ❌ Multi-user awareness ("Анна редактирует GGR прямо сейчас") — post-MVP

### Диалог с агентом

- [ ] ★ Как пользователь я хочу **спросить агента "почему просела метрика X"** и получить ответ с гипотезой, опираясь на спеку, исторические значения и прошлые аномалии `[M7, M8]`

## Post-MVP идеи (зафиксированы, не обязательство)

Эти фичи озвучены автором как "хочется", но **не для первого MVP**. Зафиксированы здесь чтобы не потерять идею.

### Architecture-relevant (могут потребовать backwards-incompatible изменений если думать о них поздно)

- **RBAC — видимость метрик по ролям.** Кейс: казино не светят финансовые метрики перед младшими аналитиками. Если когда-то будет multi-user, RBAC должен быть встроен с самого начала. **Архитектурный hint:** не закладывать решений сейчас, которые сделают невозможным introduction of роли позже (например, при сохранении anomaly-заметки запоминать `created_by` уже сейчас, даже если в MVP это всегда single user).

### UX-фичи

- **Конструктор дашборда** — pro-режим, отдельная фича. Пользователь сам собирает дашборд из карточек метрик. Большой кусок работы, отдельная разработка.
- **Импорт каталога из своей схемы** при первом запуске (option 3 в setup wizard) — в MVP это заглушка "Coming soon".

### Расширения интеграций

- **HTTP endpoint, SFTP** как дополнительные sync-методы (S3-bucket уже в MVP по решению).
- **Cloud git remote** — push/pull, разрешение конфликтов, multi-user через GitHub/GitLab/Gitea.
