## FIX-A: Подсветка путей через pinned-ноду

**Симптом:** на Graph при hover на ноду подсвечиваются связанные edges + связанные ноды (визуально выделяется путь). При unhover — подсветка исчезает. Если нода **запинена** — подсветка тоже исчезает после ухода курсора. Это убивает investigation use case: смысл pin'a в том чтобы держать в фокусе связанные пути.

**Что сделать:**
- При hover на ноду — подсветка работает как сейчас (без изменений)
- Когда пользователь **пинит** ноду (single click) — её "highlighted" состояние **сохраняется**: edges и related ноды остаются подсвеченными постоянно, пока метрика в pinned-стэке
- Если pinned ≥ 2 ноды — подсвечиваются пути для **всех** запиненных (визуально как union: edges входящие/исходящие из любой pinned-ноды + сами pinned-ноды как "выделенные")
- При unpin — подсветка для этой ноды исчезает

**Detail на твоё усмотрение:**
- Может потребоваться немного другой visual style для "pinned-highlighted" vs "hover-highlighted" — например, чуть более насыщенный цвет для pinned (потому что постоянное состояние, не временное hover-эффект). Или одинаковый цвет — оба значат "в фокусе". Решай по визуальному ощущению.
- Если две запиненные ноды связаны через путь — этот путь подсвечивается естественно (т.к. оба end'a pinned).

## FIX-B: Tooltip следует за нодой

**Симптом:** hover-tooltip на ноде сейчас рендерится в **фиксированной позиции верхнего левого угла canvas** (или похоже). Пользователю непонятно к какой ноде он относится, особенно когда ноды плотно расположены или граф заpan'ится.

**Что сделать:**
- Tooltip позиционируется **рядом с ховернутой нодой** (не у курсора, не в углу)
- Стандартный pattern: tooltip справа-снизу от ноды (с offset 8-12px), с авто-flip если нода у правого/нижнего края canvas (тогда слева-сверху)
- **Не следует за курсором** — fix относительно ноды, не mouse position. Это даёт стабильное чтение текста (если tooltip двигается с курсором — читать тяжело).
- При unhover (или при hover на другую ноду) — tooltip перемещается к новой ноде

**Технически:**
- Координаты ноды известны (она же отрисована на каноне). Tooltip позиционируется через `transform: translate(x, y)` на overlay-слое (FIX-2 первого fix-cycle разделил canvas/UI overlay — tooltip уже в overlay-слое)
- При zoom — координаты ноды масштабируются с canvas, tooltip остаётся фиксированного размера (FIX-2 поведение). Координаты pos = node-position-in-screen-space (после применения transform графа). Если у тебя SVG transform на canvas — bounding rect ноды через `getBoundingClientRect()`.

## FIX-C: Open-spec кнопка в pinned-карточке — всегда видна + restyle

**Симптом (по скриншоту):**
1. Кнопка `OPEN →` в pinned-карточке появляется **только при hover** (с поясняющим тултипом "Open spec (double-click anywhere)"). Это плохо для discoverability — пользователь не знает что кнопка есть.
2. Стиль кнопки на тёмной теме — фоном/обводкой акцентного цвета (зелёного/розового), но **выглядит чужеродно** vs ControlBar-стиль (там кнопки более минималистичные, в стиле текущего топбара).

**Что сделать:**
1. **Сделать кнопку постоянно видимой** в pinned-карточке (убрать hover-condition появления). Tooltip "Open spec" можно оставить либо убрать — на твоё усмотрение, кнопка теперь и так видна.
2. **Restyle под стиль контролов в ControlBar** — посмотри на стиль кнопок в ControlBar (Filters btn, Size selector, Zoom controls, fit). Применить тот же визуальный стиль (border, padding, hover-state). Цель — кнопка ощущается частью того же дизайн-системы, не выделяется как accent.

**Detail:**
- Двойной клик на pinned-карточку (как было) **остаётся** альтернативным способом открыть Drawer. Кнопка — explicit альтернатива.
- Если "OPEN →" текст слишком многословен в стиле ControlBar — может быть просто иконка (стрелка / open). На твоё усмотрение.
