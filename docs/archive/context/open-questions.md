## Для v0.3 Хугин

### Q3.1. Node.js или Python для MCP-сервера?

**Аргументы за Node:**
- MCP TypeScript SDK — эталонный, самый активно развивающийся
- Легко публиковать на npm
- Интеграция с фронтом (один язык всего стека)

**Аргументы за Python:**
- DuckDB-Python API очень богатый
- Личный опыт data engineering больше в Python
- Scenarios для синтетики тоже на Python — одна экосистема

**Склонность:** отложено до старта v0.3.

---

### Q3.2. Встроенный чат в UI или только Claude Desktop?

**Вариант A:** Чат виджет прямо в ODIN, через Anthropic API (нужен API key, нужна прокладка).
**Вариант B:** MCP-сервер работает только локально в Claude Desktop. В ODIN UI — инструкция "установи Хугин в Claude Desktop → открой Claude → задавай вопросы".

**Trade-off:**
- A даёт цельное UX, но требует бэкенд (ломает zero-backend архитектуру)
- B проще реализовать, но distancing: рекрутёр должен ставить Claude Desktop чтобы увидеть магию

**Компромисс:** B для v0.3, A если будет v0.4+. На v0.3 сделать хорошее 90-секундное demo-видео, где всё снято в Claude Desktop.

## Для v0.2 Мунин — MD-архитектура (новое после ADR-007)

### Q2.7. Схема frontmatter для spec.md

**Контекст:** в ADR-007 зафиксировано, что каждая метрика = MD-файл с frontmatter. Нужна схема: какие поля в yaml-хедере, какие — в body.

**Черновик (требует обсуждения):**

```yaml
---
id: ggr_margin
title: GGR Margin / Hold %
category: Revenue
level: L1
priority: Must
frequency: Daily
owner: Product
source: BI
formula: GGR / Total Wager × 100%
numerator: GGR
denominator: Total Wager
benchmarks:
  tier1: Casino 3-5%, Sports 6-8%
  latam_cis: Casino 3-5%, Sports 5-7%
  india_sea: Casino 3-5%, Sports 5-7%
thresholds:
  red: "<2% casino"
  yellow: "<3% casino"
dimensions: [geo, продукт]
dependencies: [GGR, Total Wager (Casino)]
---

# Важность

Процент удержания — сколько от каждой ставки остаётся оператору...

# Ловушки

1) Процент удержания казино (3-5%) и спорта (5-8%) нельзя сравнивать...
2) На коротком периоде ...
3) Попытка поднять hold%...

# Как улучшать

1) Смещать набор игр...
2) В спортсбуке: настраивать маржу...
```

**Открытые вопросы:**
- Что в frontmatter (structured), что в body (prose)? Склонность: короткие атомарные поля → frontmatter; длинные списки и нарратив → body.
- Ловушки и "как улучшать" — сохранять как numbered list в body, или вытащить в structured список во frontmatter? Склонность: body, чтобы легко редактировать в любом MD-редакторе.
- Как кодировать wiki-links на зависимости: `[[GGR]]` в prose или `dependencies: [ggr]` во frontmatter? Скорее всего — оба, они служат разным целям.

### Q2.8. Anomaly-заметки — формат

**Контекст:** аномалии предлагается логировать как отдельные MD в папке метрики (`metrics/<metric_id>/anomaly-YYYY-MM-DD-*.md`).

**Черновик:**

```yaml
---
type: anomaly
metric_id: ggr_margin
date_detected: 2026-03-15
date_resolved: 2026-03-22
severity: red
dimensions:
  geo: br
  product: casino
root_cause: bonus_abuse
resolved_by: analyst@example.com
---

# Что произошло

15 марта GGR Margin по Casino в Бразилии упал с 3.5% до 1.8%...

# Что выяснилось

Промо-кампания привлекла bonus abusers. См. FTD Count в BR
за тот же период — рост на 40% без изменений Retention D7.

# Ловушка #2 сработала

(ссылка на ловушку в spec.md)

# Как починили

Отозвали промо-код, добавили дополнительную антифрод-проверку...
```

**Открытые вопросы:**
- Обязательные поля vs опциональные? Склонность: обязательны `type, metric_id, date_detected`; всё остальное — по мере расследования.
- Хугин сам создаёт anomaly-заметки или только читает? v0.3 скорее всего **только читает**. Создание (`log_anomaly` tool) — v0.4+.
- Нужна ли отдельная папка `anomalies/` или привязка к метрике через frontmatter? Склонность: в папке метрики — проще контекст, проще wiki-links.

### Q2.9. Миграция из current JSON в MD-файлы

**Контекст:** у нас уже есть `igaming_metrics_normalized.json` с 236 записями. Нужен скрипт конверсии JSON → 236 MD-файлов.

**Открытые вопросы:**
- Разово (один скрипт, результат коммитим в репо) или постоянно (JSON → MD на каждом билде)? Склонность: разово. После миграции JSON как источник выбрасывается, MD становится source of truth.
- Как сгенерировать `id` каждой метрики? Склонность: slug из названия ("GGR Margin / Hold %" → `ggr_margin_hold_pct`).
- Куда девать оригинальные русские и английские названия, тэги и пр.? Частично в frontmatter, частично в body.

### Q2.10. Как UI будет подхватывать обновления из MD

**Варианты:**
- **A. Build-time:** при билде GitHub Pages читаем все MD, генерим bundled JSON, UI читает JSON. Простой, быстрый read. Но каждое изменение MD = rebuild.
- **B. Runtime:** UI при загрузке делает fetch каждого MD-файла по нужде. Медленнее, но ближе к "живой wiki".
- **C. Гибрид:** bundled index (id + frontmatter-поля для фильтров) + runtime fetch body при открытии карточки.

**Склонность:** **C (гибрид)**. Фильтры работают быстро (всё из index), полный body тянется только когда пользователь открывает конкретную карточку.

## Архитектурные вопросы (после переосмысления концепции)

### QA.1. Electron vs Tauri

**Контекст:** обе технологии решают одну задачу — упаковать React-приложение в desktop-app с доступом к файловой системе.

**Electron:**
- Плюсы: проверенный паттерн (Obsidian, VS Code, Slack), огромная экосистема, проще онбординг
- Минусы: ~150MB установщик, Chromium + Node внутри
- Бэкенд: Node.js (тот же стек что фронт)

**Tauri:**
- Плюсы: ~10MB установщик, нативный системный WebView, быстрее
- Минусы: бэкенд на Rust (новая прослойка), системный WebView = разное поведение Mac/Windows
- Меньше экосистема, новее

**Склонность:** Electron для MVP. У автора уже опыт с Obsidian (тоже Electron), F12 показывает обычный Chromium DevTools. Tauri — кандидат на будущую "lightweight edition", если станет важен размер инсталлятора.

**Когда решать:** в начале M3 (Electron-обёртка). Можно стартовать с Electron, переезд на Tauri в будущем — middle effort, не блокер.

---

### QA.2. Node.js vs Python для MCP-сервера

См. open-questions.md (выше) Q3.1.

После ADR-008 (single-process Electron, MCP — child process) аргументы перебалансировались:

**Node.js:** легче интеграция с main process Electron, общий package manager, единая экосистема. **Сильный аргумент после ADR-008.**

**Python:** удобнее для синтетики (которая всё равно на Python), DuckDB-Python API очень богатый.

**Склонность:** **Node.js** — он лучше встраивается в Electron как child process. Python остаётся для генерации синтетических данных (это offline-этап, не runtime).

### QA.3. Default LLM для MVP

**Контекст:** какую локальную модель рекомендовать в дефолте?

**Кандидаты на железе автора (32GB RAM, RTX 4060 8GB VRAM):**
- **Qwen 2.5 14B Q4** — хороший баланс, влезает в VRAM, неплохо рассуждает на русском/английском
- **Llama 3.1 8B Q4** — быстрее, проще, чуть слабее
- **DeepSeek R1 Distill 14B** — может быть сильнее в reasoning, но новее, меньше тестов
- **Phi-4 14B** — компактный, заточенный на reasoning

**На сильном железе пользователей:**
- **Llama 3.1 70B Q4** — нужно 40+GB VRAM или мощный CPU
- **Qwen 2.5 72B** — аналогично

**Склонность:** в дефолте — **Qwen 2.5 14B**. В UI настройки — список рекомендованных + custom URL для любого OpenAI-compatible endpoint.

**Когда решать:** в M8. До этого момента можно работать на Claude через MCP, не блокирует.

### QA.4. Vault location strategy

**Контекст:** ODIN работает с папкой ("vault") на машине пользователя. Как выбирается её путь?

**Варианты:**
- **A. Дефолтный путь** (`~/odin-vault/`) с возможностью изменить в настройках
- **B. Выбор при первом запуске** (как Obsidian)
- **C. Multi-vault** — пользователь может работать с несколькими vault'ами (для разных компаний, для разных проектов)

**Склонность:** **B для MVP**, **C на потом**. Multi-vault — типовая Obsidian-фича, можно добавить когда встретится реальный кейс.

**Когда решать:** в M3 (Electron-обёртка) — там это первый touch point.

### QA.5. Bundling default vault со 236 метриками

**Контекст:** при первой установке ODIN, у пользователя пустой vault. Откуда возьмутся 236 метрик?

**Варианты:**
- **A. Default vault включён в bundle ODIN.** При первом запуске — копируется из ресурсов приложения в выбранную пользователем папку.
- **B. Скачивание отдельным шагом.** Пользователь может выбрать "fresh vault" или "load default catalog".
- **C. Загрузка с GitHub.** ODIN при первом запуске тянет каталог из публичного репозитория.

**Склонность:** **A** — bundling. Пользователь сразу получает работающий каталог. Custom-метрики — добавляет руками, если хочет.

**Trade-off:** размер инсталлятора растёт на ~5MB (236 MD-файлов + синтетика). Незначительно vs total size Electron.

**Когда решать:** в M2 (миграция в MD).
