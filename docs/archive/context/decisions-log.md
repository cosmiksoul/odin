## ADR-001: Упрощение концепции до трёх слоёв

**Дата:** 2026-04-24
**Статус:** принято

**Контекст:**
Изначальная идея проекта iGaming Intelligence MCP была шире — включала внешние источники (regulatory, stocks, prediction markets, competitor pulse). При обсуждении стало понятно, что такой скоуп — 4-5 месяцев вечерами, риск заморозки уже готового Catalog на полгода.

**Решение:**
Упростить до трёх слоёв: ODIN (фронт/каталог) + Мунин (DWH) + Хугин (MCP). Всё остальное — out of scope.

**Последствия:**
- Публикация v0.1 возможна уже сейчас
- Старая идея iGaming Intelligence MCP — отдельный потенциальный проект, в скоуп ODIN не входит
- Проект становится фокусированным: "фэнтези-BI с LLM-интерфейсом", а не "индустриальная разведка"

## ADR-002: Пошаговая публикация v0.1 → v0.3

**Дата:** 2026-04-24
**Статус:** принято

**Контекст:**
Два варианта — публиковать всё сразу через 5 месяцев или по частям.

**Решение:**
Публиковать тремя итерациями: v0.1 (Catalog), v0.2 (+ Мунин), v0.3 (+ Хугин).

**Аргументы:**

**Компромисс:**
В v0.1 имя ODIN и концепция "бог и два ворона" упомянуты (в About), но функционально работает только Catalog. Это создаёт лёгкую тизер-рамку для будущих версий.

## ADR-003: DuckDB + Parquet вместо BigQuery / Supabase

**Дата:** 2026-04-24
**Статус:** принято

**Контекст:**
Для Мунина рассматривались: BigQuery, Supabase (PostgreSQL), ClickHouse self-hosted, DuckDB + Parquet в репо.

**Решение:**
DuckDB + Parquet.

**Аргументы за:**
- Zero-backend — совместимо с GitHub Pages
- DuckDB-WASM работает в браузере, данные доступны фронту напрямую
- Тот же Parquet читает MCP-сервер через DuckDB embedded — единый data layer
- Миграция на "настоящий" DWH в будущем тривиальна (SQL стандартный, типы сохраняются)
- Паттерн соответствует продакшн-реальности: "Мунин — реплика с приведением к плоскому виду"
- Размер данных маленький (40-80 MB) — BQ был бы overkill

**Аргументы против (трейд-оффы):**
- Обновление данных = git commit (для демки норм, для прода нет)
- Нет write-path для пользовательских данных (комментарии, сохранённые фильтры) — если захочется, потребуется отдельное хранилище
- Первоначальная загрузка 40-80 MB — решается через HTTP range requests и разбиение на чанки

## ADR-004: Мастер-view в long format

**Дата:** 2026-04-24
**Статус:** принято (подтверждено автором 2026-04-25)

**Контекст:**
Выбор формата хранения значений метрик с разрезами.

**Решение:**
Long format: одна строка на `(metric × date × dim_key × dim_value)`.

**Аргументы:**
- Гибче для добавления новых разрезов без миграции схемы
- В DuckDB по производительности разница с wide минимальна
- Проще для MCP-запросов (единый паттерн SQL)
- Совместимо со спарсенным каталогом метрик — каждая метрика идентифицируется по `metric_id`, который мы получили из исходного справочника

**Открытые вопросы:**
См. munin-master-view.md — Q1-Q4.

## ADR-007: Переосмысление Мунина — editable semantic layer вместо просто DWH-коннектора

**Дата:** 2026-04-24
**Статус:** принято

**Контекст:**
При ревью первого прототипа UI (Claude Design) возникло понимание, что текущее определение Мунина ("коннектор к DWH + flat master-view") слабо передаёт его ценность. Классические BI-инструменты проблему "знание о метрике умирает в головах команды" не решают — спеки лежат в одном месте (Notion, Confluence, dbt docs), данные в другом, а связь между ними — в head'ах аналитиков.

Если Хугин (v0.3) должен отвечать на вопросы типа "почему метрика просела?", ему нужен **институциональный контекст**: не только формула и текущее значение, но и история — какие аномалии уже были, какие гипотезы проверяли, что команда на практике узнала об этой метрике. Этот контекст должен где-то жить и должен быть редактируемым.

**Решение:**
Мунин — это **editable semantic layer**, а не просто коннектор. Архитектурно:

- **Каждая метрика = markdown-файл** с frontmatter (structured fields) + body (human-readable описание, ловушки, fix-рецепты)
- **Changelog = git history** каждого MD-файла (бесплатно, из коробки)
- **Аномалии = отдельные MD-заметки** внутри папки метрики (`anomaly-2026-03-15-ggr-br.md`) с frontmatter `type: anomaly, date, resolved_by, metric_id`
- **Связи между метриками = wiki-links** `[[GGR]]` внутри MD (как в Obsidian)
- **Данные (значения метрик) = по-прежнему Parquet** через DuckDB — но это **один из источников контекста**, не главный

Мунин теперь хранит два типа "памяти":
1. **Память о значениях** (Parquet — временные ряды) — что было
2. **Память о смысле** (MD-файлы + git history) — что мы об этом поняли

**Аргументы за:**
- Решает главную проблему BI: "знание о метрике умирает в головах"
- Хугин получает гораздо более богатый контекст — не только "что написано в спеке", но и "что команда знает об этой метрике исторически"
- MD — универсальный формат, который LLM читает напрямую без парсинга
- Git даёт versioning, rollback, blame, branching — бесплатно
- Совпадает с текущим workflow автора (Obsidian + git)
- Портфельная история сильнее: не "красивый BI с LLM-чатом", а "institutional knowledge system, превращённая в прод"
- Хранилище отделено от редактора — ODIN под капотом MD/git, а UI поверх любого вида (см. ADR-011 для решения по UI: встроенный WYSIWYG)

**Аргументы против / трейд-оффы:**
- Скоуп v0.2 растёт: нужен не только генератор синтетических данных, но и базовая архитектура метрик-как-MD-файлов
- MD хуже для табличных данных (но в карточке метрики данных и нет, только спеки)
- Git-модель непривычна для non-dev пользователей (митигация: см. подробный вариант ниже)

**Под-вариант реализации хранилища:** Мунин — это **локальная папка с MD-файлами + git** на машине пользователя. Структура: `<vault>/metrics/<metric_id>/spec.md` для спек и `anomaly-*.md` для инцидентов. Git инициализирован в vault'е, коммиты на каждое изменение.

**Локальный или shared:**
- В MVP пользователь может работать **полностью локально** (без remote), либо подключить git remote для синхронизации с командой
- Подробная стратегия — см. ADR-013
- Архитектурно vault одинаков в обоих случаях — наличие/отсутствие remote не меняет внутреннюю структуру

> Решение про **UI редактирования** (встроенный WYSIWYG vs внешний редактор) — отдельная архитектурная развилка, см. **ADR-011**. Ядро ADR-007 — про **где хранится семантика**, а не про **как пользователь её правит**.

**Последствия:**
- Роль Мунина расширяется: он не только хранит значения, но и описания, контекст, историю
- Структура репозитория меняется (см. обновлённый architecture.md)
- Миграция из текущего `igaming_metrics_normalized.json` в MD-файлы — отдельная задача (M2 в JTBD-full-scope.md)
- Каждое изменение MD-файла — атомарный git commit. История метрики = git log по её файлам.
- Хугин (MCP-сервер) читает MD-файлы напрямую — это его источник смыслового контекста.
- **Способ редактирования** — отдельное решение, см. ADR-011 (встроенный WYSIWYG в ODIN).

**Метафора (если нужно объяснить в одной фразе):**
> Мунин — это wiki команды по метрикам, которую читает и LLM, и ODIN-фронт, и сами люди. Git хранит все версии. MD — общий язык.

## ADR-008: Local-first архитектура — Electron app + опциональная локальная LLM

**Дата:** 2026-04-24
**Статус:** принято

**Контекст:**
В обсуждении вскрылись два ключевых факта, которые меняют траекторию проекта:

1. **iGaming-аудитория параноидальна по compliance.** Никакой data-team регулируемого оператора не одобрит инструмент, который отправляет когортные данные в Anthropic/OpenAI. Это не "приятная фича" — это **жёсткий блок** для реального использования.
2. **Автор лично хочет local-first MVP.** Вечерами полгода разрабатывать комфортнее как локальную apku, чем как cloud-сервис с auth/hosting/payment.

Эти два фактора **сходятся в одной точке**: правильная архитектура для ODIN — desktop-приложение с локальной обработкой данных и опциональной локальной LLM.

**Решение:**
ODIN — **локальная desktop-app** (Electron или Tauri). Все данные на машине пользователя. LLM по выбору: локальная (Ollama, LM Studio, vLLM) или Claude Desktop через MCP. Никакого cloud-хостинга в основном пути.

**Конкретные следствия:**

- **Frontend** — Electron-обёртка над React-приложением. F12 даёт DevTools, разработка как в обычном web-app, но с доступом к файловой системе.
- **Storage values** — DuckDB-файл локально (опция: Parquet + DuckDB)
- **Storage semantics** — MD-файлы в локальной папке. Версионность через локальный git (без push если пользователь так захочет)
- **MCP-сервер** — child process самого Electron-приложения. Запустил ODIN → MCP автоматически доступен Claude Desktop.
- **LLM** — bring-your-own. Поддержка Ollama / OpenAI-compatible endpoint / Claude Desktop через MCP. Дефолт: локальная модель (Qwen 14B или Llama 3.1 8B на железе автора).

**Аргументы за:**
- **Privacy-first как killer feature.** Это **отличает** ODIN от Tableau+ChatGPT plugin, Hex, любого SaaS-BI. Сильное позиционирование в регулируемых индустриях.
- **Целостное demo-UX** — одна иконка на Dock, одна программа, всё работает локально. Естественно для b2b-аудитории, где "покажи локальную демку под NDA" — стандартный mode.
- **Снимает целый класс проблем:** GitHub API rate limits, OAuth/PAT auth для **самой apки**, hosting, CORS, deployment pipeline, серверный API key для LLM, proxy endpoint, security compliance на wire.
- **Соответствует автору:** автор сам один пилит вечерами, нет нужды в multi-tenant cloud-инфраструктуре из коробки.
- **Electron не страшнее web-app.** Obsidian — Electron, F12 показывает обычный Chromium DevTools. Минимальный setup — 30-50 строк main.js плюс electron-builder. Разработка идёт как в обычном React-проекте.
- **Bring-your-own-model** — позволяет каждому запустить под свои compliance-требования. Параноикам — Ollama. Тем, кому можно cloud — Claude Desktop.

**Что именно запрещено (уточнение, чтобы не съесть нужные кейсы):**

1. ❌ **Cloud OAuth для самой apки.** Нет "Login with Google" чтобы войти в ODIN, нет SaaS-аккаунтов, нет серверной авторизации.
2. ❌ **Hosted backend от Anthropic/нашей стороны.** Нет Vercel/Railway/Fly.io как обязательной инфраструктуры в основном пути. Нет нашего API-сервера между ODIN и LLM.
3. ❌ **Multi-tenant cloud architecture.** Один пользователь = одна установка. Нет учёток в облаке.

**Что НЕ запрещено (важно):**

1. ✅ **Локальный credentials store** — для S3-ключей, API-токенов LM Studio, токенов к корпоративному git remote. Хранится в OS-native secret store (macOS Keychain, Windows Credential Manager, Linux secret-tool). См. ADR-012.
2. ✅ **Push/pull к git remote** — пользователь может настроить remote git (GitHub/GitLab/корпоративный Gitea) для синхронизации vault'а с командой. См. ADR-013.
3. ✅ **Внутренний RBAC** в будущем — если когда-то будет multi-user в одном vault'е, роли управляются ODIN на уровне frontmatter метрик, без cloud auth provider. Post-MVP.
4. ✅ **Опциональные cloud LLM** через user-supplied credentials (Anthropic API key, OpenAI key) — пользователь сам решает, его ключи в credentials store.

**Аргументы против / трейд-оффы:**
- **Локальная модель хуже Claude** для сложных аналитических задач. Llama 3.1 8B путается в многошаговых рассуждениях. 70B нужен мощный ноутбук. Митигация: dual-mode — по умолчанию локальная, переключение на Claude когда policy позволяет.
- **Distribution как .dmg/.exe** — пользователь должен скачать и установить. Не "ткни в ссылку и работает". Для b2b это норма, для casual show-off — минус.
- **Размер инсталлятора** — Electron ~150MB, Tauri ~10MB. Tauri привлекательнее по размеру, но требует Rust-прослойки. Решение между ними — отложено до старта реализации.
- **Нет "просто посмотреть в браузере"** — для рекрутёров/casual viewers невозможно увидеть продукт без установки. Митигация: demo-видео 90 секунд + опционально публичная read-only демка как отдельный артефакт.

**Последствия:**
- **Vercel/GitHub Pages из стека выпадают** для основного пути. Возможно остаются для отдельной публичной демки (если делать).
- **GitHub API integration выпадает** из основного пути — git локальный. Если у пользователя настроен git remote — push/pull через стандартный git protocol, не через GitHub REST API.
- **Cloud OAuth для входа в apку — out of scope** для всех v0.x. Внутренний RBAC и multi-user — post-MVP с архитектурным hint в коде.
- **Distribution channel** — GitHub Releases с .dmg и .exe. Установщик — electron-builder.
- **MCP-сервер архитектурно меняется** — теперь это child process Electron, не отдельная установка через npx.
- **Демонстрация продукта** требует записанного demo-видео. Без видео не показать что инструмент умеет.
- **Credentials store нужен в MVP** — для S3-ключей, API-токенов, git remote credentials. См. ADR-012.
- **Git remote sync — опциональная фича MVP.** Поэтапная реализация: single-user локально → push/pull к remote → conflict resolution → multi-user с RBAC. См. ADR-013.
- Новое позиционирование: *"ODIN — локальный BI-ассистент для iGaming. Ваши данные никуда не уходят. Ваша модель — любая, какую поставите. Ваша команда — через ваш git."*

## ADR-009: ~~MVP-цель — сценарий 3 (Хугин-демо), редактор out of MVP scope~~ ОТОЗВАНО

**Дата:** 2026-04-24
**Дата отзыва:** 2026-04-25
**Статус:** ❌ ОТОЗВАНО — заменено ADR-011

> ⚠ **Это решение было ошибочным и отозвано на следующий день.** Сохранено в логе для прозрачности рассуждений.

**Что было решено (ошибочно):**
Главная цель MVP — работающий сценарий 3 (Хугин-демо). In-app WYSIWYG редактор и history UI — out of MVP scope, отложены до v0.5+.

**Почему это было ошибкой:**

При обсуждении скоупа MVP я (Claude) **подменил продуктовую логику демо-логикой**:
- Сравнивались три demo-сценария для портфолио. Сценарий 3 (Хугин) был выбран как самый сильный для 90-секундного видео.
- Из этого выбора я вывел: "редактор не нужен для демо → отложим из MVP".
- Это противоречило **прямому продуктовому требованию автора**, высказанному ранее:
  > *"мы делаем инструмент для аналитиков. из опыта аналитикам сложно объяснить даже концепцию гита... они не совсем инженеры. У меня есть опыт внедрения инструментов code-as-service в аналитические группы — это пиздец, сопротивление с нулевой минуты"*

Это сильное утверждение из реального опыта внедрения, а не теоретическая боль. Я ответил тогда "в этом ты меня переубедил" — и потом всё равно потерял это в обсуждении скоупа.

**Урок для будущих ADR:**
- **Демо-сценарий ≠ продуктовый скоуп.** Демо показывает что инструмент умеет в идеальных условиях. Продукт должен работать в реальных условиях с реальной аудиторией.
- **Когда автор называет продуктовую боль из опыта — она не отменяется** аргументами "для демо это не критично".
- При выборе "что в MVP" приоритет: **что нужно пользователю**, а не **что красивее в видео**.

**Замещающее решение:** см. ADR-011.

## ADR-011: In-app WYSIWYG-редактор и history UI — обязательная часть MVP

**Дата:** 2026-04-25
**Статус:** принято
**Заменяет:** ADR-009 (отозвано)
**Уточнено:** ADR-016 (2026-04-27) — WYSIWYG обязателен в **MVP-2**, не в MVP-1. MVP-1 — read-only Catalog без редактора (это промежуточная веха, не позиционируется как «инструмент для аналитиков-редакторов»).

**Контекст:**

ICP проекта — аналитики и продакты в iGaming/финтех/healthcare. Это **профессионалы по работе с данными**, но не инженеры. Многие из них не знают git на уровне commit/branch/merge, не работали с MD-разметкой, не используют CLI.

Опыт автора по внедрению code-as-service инструментов в аналитические команды показывает: **сопротивление возникает с первого дня** если инструмент требует git/MD-знаний. Чем больше "программистских" зависимостей — тем меньше реальное использование.

Если ODIN требует от пользователя:
- Установить VS Code или другой MD-редактор
- Понять frontmatter в YAML
- Знать `git commit`, `git push`, `git log`, `git revert`
- Разобраться как разруливать конфликты

— то **большинство целевой аудитории не дойдёт до использования продукта**, даже если технически он установлен.

**Решение:**

In-app WYSIWYG-редактор и history UI — **обязательная часть MVP**. ODIN должен быть полностью самодостаточен для пользователя-аналитика. Никаких внешних редакторов, никакого git в UX.

**Конкретно:**

**WYSIWYG-редактор:**
- Кнопка "Edit" в карточке метрики
- Frontmatter поля рендерятся как формочки (dropdown для уровня, input для формулы, и т.д.)
- Body (важность, ловушки, как улучшать) — rich-text редактор (TipTap / Lexical / ProseMirror)
- Кнопка "Save" с индикатором статуса ("Сохранено", "Сохраняется...")
- Под капотом: сохранение в MD-файл с frontmatter + локальный git commit с автогенерируемым сообщением (например, "Updated GGR Margin formula")

**History UI:**
- Кнопка "History" в карточке метрики
- Список версий с человеческими сообщениями ("Анна, 12 мая, 14:32 — изменила формулу")
- Каждая версия → "Посмотреть" (показывает diff в human-readable виде, не raw `+/-`) и "Вернуть эту версию" (создаёт новый коммит с откатом)
- Под капотом: рендеринг из git log + git diff, но в UX нет ни слова про коммиты

**Что НЕ нужно делать в UX:**
- Нет упоминания git, веток, коммитов
- В рамках одной редактирующей сессии нет понятия "merge conflict" — конфликты возникают только при pull от коллеги (см. ADR-013), они отдельный flow
- Нет команды "push" в кнопке Save — push выполняется отдельно через Settings → Vault sync (ADR-013), либо автоматически по настройке

**Аналогии:** Notion для редактирования, Google Docs для history. Аналитики знают эти паттерны, они комфортные.

**Аргументы за:**
- **Адресует реальную боль ICP.** Без этого ODIN — инструмент для разработчиков, а не для аналитиков (несмотря на позиционирование).
- **Privacy-first консистентность.** Раз ODIN — local-first apka (ADR-008), то и редактирование должно быть в самой apke. "Установи ещё VS Code и научись git" — противоречит UX обещанию "одна иконка, всё работает".
- **Проверенный паттерн.** Obsidian работает поверх MD-файлов с локальным git, и аналитики (нетехнари) с ним справляются. ODIN может пойти тем же путём, но сделать UX ещё проще (без сырого MD-синтаксиса).
- **Под капотом всё то же.** ADR-007 (MD-файлы + git) **сохраняется** — мы не меняем хранилище, только добавляем UI поверх него. MCP-сервер (Хугин) продолжает читать MD напрямую, ничего не ломается.

**Аргументы против / трейд-оффы:**
- **Скоуп MVP растёт значительно.** WYSIWYG + history UI — это отдельная нетривиальная задача (2-4 недели работы при наличии прототипа). Митигация: автор готов пилить полгода вечерами, темп не критичен.
- **Технически сложнее.** TipTap/Lexical нужно интегрировать с frontmatter-схемой; commit messages нужно автогенерировать по diff; rollback нужно делать через новый коммит, не `git reset`. Митигация: каждый из этих кусочков посилен по отдельности, идём пошагово.
- **Спрос неподтверждён.** Автор делает MVP по сути для себя + потенциального ICP. Не факт что реальные аналитики появятся как пользователи. Контр-аргумент: даже если использовать самому, редактор удобнее чем VS Code + git CLI.

**Последствия:**

- **Roadmap меняется:** добавляется новый milestone (M-EDIT) между M5 (sync) и M6 (anomaly-заметки) — реализация WYSIWYG + history. Может быть разделён на 2-3 спринта.
- **JTBD сохраняет** оригинальные user stories автора про редактор и version control (которые я ошибочно собирался убрать).
- **CLAUDE.md для проекта** должен явно прописать: "редактор и history — часть MVP, не post-MVP".
- **Технический выбор библиотеки редактора** становится открытым вопросом — TipTap vs Lexical vs ProseMirror. Откладывается до старта реализации M-EDIT.
- **Структура frontmatter** (open-questions.md Q2.7) становится **критически важной до старта M-EDIT**. WYSIWYG не может работать с raw YAML, нужны typed-fields.

**Метафора:**
> ODIN относится к git и MD так же, как Obsidian относится к git и MD: всё под капотом, в UX — простые карточки и кнопка Edit.

## ADR-010: Push-based интеграция с DWH через Parquet-витрину

**Дата:** 2026-04-24
**Статус:** принято

**Контекст:**
Если ODIN — локальная apka, остаётся вопрос: откуда берутся данные, когда пользователь хочет работать с реальными метриками своей компании? Два варианта:

1. **Pull-based:** ODIN подключается к ДВХ компании через коннекторы (Snowflake, BigQuery, ClickHouse, etc) и сам тянет данные. Стандарт SaaS-BI.
2. **Push-based:** Компания готовит витрину в заданном формате и кладёт в место, откуда ODIN её читает. Стандарт on-premise / регулируемых индустрий.

Pull-based требует поддержки множества коннекторов, security-сертификации (DPA, SOC 2), credentials management, network access от ODIN к prod-системам компании. Это **огромный скоуп** и **жёсткая security-проблема** для iGaming.

**Решение:**
Push-based интеграция через стандартизированную Parquet-витрину. Спецификация формата документируется отдельно — см. munin-sync-contract.md.

**Контракт (high level):**
- Компания готовит файл `master_view.parquet` в схеме, заданной ODIN: `metric_id × date × dim_key × dim_value × value`
- ODIN конфигурируется на источник (локальный файл, HTTP URL, S3-bucket, watch folder)
- Sync — на расписании (раз в час / день / по триггеру) или вручную через UI
- При первом sync — валидатор: какие `metric_id` присутствуют, какие ожидаются, что отсутствует. Понятное сообщение пользователю.

**Зона ответственности:**
- **ODIN не знает** про ДВХ компании, dbt-модели, ETL, источники, security-протоколы
- **Data team компании** строит витрину (один dbt-model или SQL-запрос) и кладёт в указанное место
- **ODIN читает** Parquet и работает с тем что пришло

**Аргументы за:**
- **Никакого ETL в ODIN.** Не пишем коннекторы к Snowflake/BigQuery/etc. Не поддерживаем 20 разных диалектов SQL.
- **Никакого доступа к продакшену компании.** ODIN не ходит в их системы. Они сами кладут файл.
- **Security принимает.** Отправить Parquet по защищённому каналу — повседневная практика data-команд. Не нужны DPA, SOC 2, network access.
- **Обновление данных = их ответственность.** Запланируй cron, обнови файл. Сломалось — их data team разбирается.
- **dbt-style контракт.** Знакомый паттерн для data-инженеров. Они сразу понимают: "нам нужна одна витрина в их схеме, как dbt-модель".
- **Гибкость:** компания может фильтровать что отдавать (только publishable метрики, только определённые ГЕО, маскированные значения).

**Аргументы против / трейд-оффы:**
- **Барьер для casual использования.** Аналитик не может "просто открыть ODIN и подключиться к нашему DWH в один клик". Нужен инженер для построения витрины. Митигация: для ICP (регулируемые индустрии) это **не баг, а фича** — security-команда хочет такого контроля.
- **Контракт должен быть железно понятный.** Иначе техподдержка съест время. Митигация: подробная документация, примеры dbt-моделей, валидатор с понятными ошибками.
- **Версионирование контракта** — если ODIN v2 добавит новую колонку в схему витрины, что делать со старыми витринами? Решается forward-compatibility (новые поля nullable, старые остаются).

**Последствия:**
- **Слой 1 (storage values) упрощается** — нет нужды в множестве коннекторов. Только Parquet reader через DuckDB.
- **Появляется новая концепция:** "Munin Sync Contract" — спецификация формата витрины. Документируется в отдельной заметке munin-sync-contract.md.
- **Sync mechanisms:** для MVP достаточно "manual import через UI" + "watch local folder". Для v1+ — HTTP endpoint и S3.
- **В demo-видео можно показать:** "вот пользовательский ODIN. Вот директория, куда data-team нашей выдуманной компании положила витрину. ODIN её подхватил. Данные не покинули компанию."

## ADR-012: Credentials store — локальный безопасный кошелёк секретов

**Дата:** 2026-04-25
**Статус:** принято

**Контекст:**

С добавлением sync-механизмов (S3, HTTP, git remote) и потенциальной поддержки cloud LLM (Anthropic API key, OpenAI API key) ODIN должен где-то хранить credentials пользователя. Без этого работа с реальными источниками невозможна.

Варианты были:
- Открытый текст в config.json — небезопасно, в репо может попасть случайно
- Шифрование в config.json с master-password — добавляет UX-трение (при каждом запуске вводить пароль)
- OS-native secret store — стандартный безопасный паттерн (Keychain / Credential Manager / secret-tool)

**Решение:**

ODIN использует **OS-native secret store** для всех credentials:
- **macOS:** Keychain Access (через `keytar` библиотеку)
- **Windows:** Credential Manager (тоже `keytar`)
- **Linux:** secret-tool (libsecret) или KWallet / GNOME Keyring

Это **полнофункциональный credentials store с MVP** — не "сначала локальный Parquet, потом всё остальное". Если делаем — то сразу нормально, потому что добавлять секреты по одному = постоянные миграции.

**Что хранится:**

1. **Sync источники витрины:**
   - S3-bucket access keys (для приватных бакетов)
   - HTTP basic auth / bearer tokens
   - SFTP keys

2. **LLM credentials:**
   - Anthropic API key (если пользователь хочет cloud Claude)
   - OpenAI API key
   - LM Studio auth token (если их endpoint требует)
   - Custom OpenAI-compatible endpoint credentials

3. **Git remote credentials:**
   - SSH-ключи для git push/pull (или ссылки на существующие в `~/.ssh/`)
   - Personal Access Tokens для HTTPS git remote (GitHub PAT, GitLab token)

**Что НЕ хранится:**

- ❌ Пароли пользователя для входа в саму ODIN (в MVP вход не требуется — single-user локально, ADR-008)
- ❌ Anthropic OAuth tokens — мы не делаем OAuth flow (ADR-008)
- ❌ Данные метрик и значения — это не secrets, это контент

**Архитектура:**

- **Library:** `keytar` (npm package, обёртка над OS-native API)
- **Service name:** "ODIN" (одно пространство для всех ключей пользователя)
- **Account name:** semantic key, например `aws.s3.<bucket-name>.access_key`, `llm.anthropic.api_key`, `git.remote.<vault-name>.token`
- **UI:** Settings → Credentials → список настроенных учёток (без отображения самих секретов, только статус "Set / Not set" и имя)
- **Secret rotation:** простое — пользователь удаляет старую запись, добавляет новую

**Аргументы за:**
- **Стандартный паттерн** — пользователи привыкли к Keychain/Credential Manager (там же лежат пароли от других apок)
- **Безопасно из коробки** — OS защищает свой secret store, encryption at rest бесплатно
- **Никакой нашей реализации** — `keytar` library проверена, не изобретаем cryptographию
- **Совместимо с corporate policy** — security-команды одобряют использование OS-native стора
- **Cross-platform** — `keytar` работает на всех трёх платформах одним API

**Аргументы против / трейд-оффы:**
- **`keytar` — Node native module** — нужен rebuild при каждом обновлении Electron/Node. Митигация: electron-builder автоматизирует, в проде это не проблема.
- **Linux вариативность** — на Linux secret-tool не всегда установлен. Митигация: graceful fallback на encrypted файл с master-password.
- **Headless режим** — если пользователь хочет запустить ODIN без UI (например для CI), credentials нужны как переменные окружения. **Out of MVP scope** — UI-driven сетап достаточно.
- **Backup/restore** — credentials не попадают в backup vault'а (они в Keychain, не в файлах). При переезде на новый компьютер нужно вручную перенастроить. Митигация: документация, в будущем — export/import функция.

**Последствия:**

- В коде появляется обёртка `lib/credentials/` поверх `keytar`
- В UI появляется Settings → Credentials экран (часть M3 Electron-обёртки или отдельный milestone)
- Sync-механизмы (M5) и LLM-интеграция (M8) **зависят от credentials store** — без него не реализуются
- При деинсталляции ODIN credentials остаются в Keychain — пользователь должен их очистить вручную (как для любой apки). Документируем.

**Открытые вопросы:**
- Нужен ли master password для дополнительной защиты, помимо OS-уровня? Скорее всего нет — OS Keychain уже защищён логином пользователя.
- Как обрабатывать ситуацию когда Keychain/Credential Manager недоступен (Linux без secret-tool, headless)? Решение откладывается до фактической реализации.

## ADR-013: Shared vault через git remote — поэтапная стратегия

**Дата:** 2026-04-25
**Статус:** принято

**Контекст:**

Концепция Карпатого ("LLM пишет в общую wiki, каждый ответ компаундит знание команды") применима к ODIN: **семантический слой команды должен быть shared**, чтобы:
- Аналитики видели спеки и anomaly-заметки коллег
- Хугин писал инсайты в общую базу
- Институциональная память накапливалась команде в целом, а не индивидуально

Текущий ADR-007 (Мунин = MD + git) описывает **локальный** git. Для shared vault нужен **remote git** + механика синхронизации.

Однако полноценная реализация shared vault с auto-write от LLM, conflict resolution, RBAC — это **большой кусок продукта** (Obsidian Sync разрабатывали годами). Запихивание всего в MVP **утроит скоуп** и убьёт темп.

**Решение:**

**Поэтапная реализация** shared vault:

### Этап 1 (MVP): Single-user + push/pull к git remote

**Что включено в MVP:**
- Пользователь может **подключить vault к git remote** (GitHub, GitLab, Gitea, корпоративный bare git) через UI
- **Manual push/pull** через кнопки в Settings → Vault sync
- **Auto-pull при старте** ODIN (опция)
- **Auto-push при save** (опция, по умолчанию off — чтобы не спамить commit'ами в shared вику)
- **Стандартные git credentials** через credentials store (см. ADR-012): SSH-ключи или PAT
- **Conflict resolution = git default behavior** — если конфликт, ODIN показывает понятное сообщение "ваш коллега изменил эту метрику, разрешите конфликт через git CLI или подождите следующего pull". В MVP **мы не пишем UI разрешения конфликтов**.

**Что НЕ включено в MVP:**
- ❌ Auto-write Хугина в shared vault — **в MVP Хугин предлагает сохранить, человек подтверждает кнопкой "Save as anomaly note"**. Полный автомат — Этап 3.
- ❌ Real-time sync (как Obsidian Sync) — нет.
- ❌ Conflict resolution UI внутри ODIN — нет.
- ❌ RBAC (видимость метрик по ролям) — нет.
- ❌ Multi-user awareness ("Анна сейчас редактирует GGR") — нет.

**Архитектурный hint в коде MVP** (важно для будущих этапов):
- Anomaly-заметки записывают `created_by: <user_name>` (имя берётся из git config или Settings)
- Spec-файлы при коммите записывают autor в commit message
- Если RBAC появится позже — `created_by` уже будет, не нужна миграция

### Этап 2 (post-MVP): Conflict resolution UI + Хугин-предложения

- UI разрешения конфликтов внутри ODIN (показать обе версии, выбрать или сметчить)
- Auto-pull чаще (раз в N минут, не только при старте)
- Хугин предлагает **draft anomaly-заметки** при анализе (но человек одобряет до commit)

### Этап 3 (future): Auto-write Хугина + RBAC

- Хугин **автоматически** пишет инсайты в shared vault при определённых сценариях (после расследования аномалии, например)
- Roles: viewer / editor / admin per metric или per category
- Audit log "что Хугин написал и когда"

**Аргументы за поэтапность:**
- **Time-to-first-publish важнее completeness.** За 6 месяцев можно сделать рабочий single-user + git remote. За 18 — может не хватить терпения.
- **Реальный фидбэк > предположения.** Этапы 2 и 3 проектируем после того как этап 1 поработает у реальных пользователей. Возможно Хугин-auto-write окажется ненужным или нужным иначе.
- **Карпатый-паттерн не доказан в team-сценарии.** Карпатый показывает personal wiki, не team. Возможно в команде нужно по-другому. Лучше сначала собрать данные.
- **Conflict resolution — отдельный продукт.** Делать его UI правильно (3-way merge, semantic diff, undo) = месяцы работы. В MVP fallback на git CLI приемлем для технических пользователей. Аналитики через UI пока не редактируют конфликты — но подождут до Этапа 2.

**Аргументы против:**
- **Команда может фрустрироваться** на этапе 1 без conflict UI. Митигация: документация "как разрешить конфликт", и на практике для small team (3-5 человек) с разделением метрик по owner'ам конфликты редки.
- **Хугин без auto-write — менее впечатляющая фича.** Митигация: для demo-видео достаточно "Хугин предложил, я кликнул Save". Auto-write — wow-feature для v2.0.

**Последствия:**

- В JTBD появляются stories для git remote sync (см. обновлённую группу 7 / cross-cutting)
- В roadmap появляется milestone **M-SYNC** (между M5 и M-EDIT) — git remote integration
- Frontmatter anomaly-заметок включает `created_by` уже в M2 (миграция в MD), даже если в MVP это всегда один пользователь — **архитектурный hint для RBAC**
- ADR-007 косвенно расширяется: Мунин теперь **может быть локальным или shared** (через remote git), но архитектурно одинаков
- Хугин (M7) получает интерфейс "save as anomaly note", но не auto-write
- В CLAUDE.md обновляется список разрешённого/запрещённого: git remote OK, auto-write Хугина — нет (в MVP)

**Открытые вопросы (откладываются):**
- Какой default git provider рекомендовать в setup wizard'е? GitHub удобнее, но не все компании его используют. Возможно "укажи свой URL" без рекомендаций.
- Как обрабатывать case "vault слишком большой для git" (например, через год там тысячи anomaly-заметок и Parquet с историей)? Скорее всего git LFS для Parquet. Финализируем при реализации.
- Наследование настроек git между членами команды (один задал, остальные подхватили)? Out of MVP.

## ADR-014: Compact-карточки метрик — отдельный компонент, не вариант MetricPreview

**Дата:** 2026-04-27
**Статус:** принято
**Контекст возникновения:** Sprint 2 fix-cycle 1 (восстановление multi-pin в Graph view)

**Контекст:**

В Sprint 2 Pass 2 был сделан унифицированный `MetricPreview` — общий компонент для drawer Catalog и drawer Graph. В рамках этого решения Code удалил multi-pin side-panel из Graph (как «дублирующий source of truth»). Это оказалось ошибочным продуктовым решением — multi-pin критичен для investigation use case в Graph (side-by-side сравнение запиненных метрик).

При восстановлении multi-pin в fix-cycle 1 встал вопрос как реализовать compact-карточку в pinned-stack:

- **Вариант A:** добавить `compact` flag в `MetricPreview` — переиспользовать тот же компонент с уменьшенными paddings, сворачиваемыми секциями, без полного body
- **Вариант B:** сделать отдельный компонент `PinnedMetricCard` с собственной разметкой и стилями, независимо от `MetricPreview`

Code выбрал **B** (PinnedMetricCard как отдельный компонент). Это компонентное архитектурное решение, повторяемое для будущих compact-видов карточек (Saved view, потенциальный Home dashboard, hover-card со спарклайном).

**Решение:**

**Compact-виды метрик в UI = отдельные компоненты, не варианты `MetricPreview`.**

`MetricPreview` остаётся **single-purpose** — drawer для full view метрики (Catalog и Graph). Compact-форматы (pinned-карточка, summary в Saved view, и т.п.) реализуются как самостоятельные компоненты, переиспользующие визуальные примитивы (status badge, sparkline, value formatting) из `src/lib/util.js`.

**Аргументы за (вариант B):**

- **Single Responsibility.** `MetricPreview` не разрастается conditional-логикой (`if (compact) { ... } else { ... }`). Каждый компонент решает одну задачу.
- **Независимое тюнинг compact-формата.** Pinned-карточка должна быть компактной, читаемой при двух-четырёх запиненных метриках, помещаться в side-panel шириной ~280px. Эти constraints не пересекаются с full drawer.
- **Безопаснее для рефакторинга.** Изменения в drawer не ломают pinned-stack, и наоборот.
- **Прозрачнее для нового разработчика.** Имя `PinnedMetricCard` говорит назначение; «MetricPreview compact» требует чтения кода.
- **Соответствует прецеденту.** Catalog drawer и Graph drawer уже разделили ответственность с общим `MetricPreview` — здесь следующий шаг логичен: ещё один контекст (pinned), ещё один компонент.

**Аргументы против / трейд-оффы:**

- **Визуальный дрифт.** Если в `MetricPreview` меняется style карточки (например, padding, border-radius, цвет status-badge) — `PinnedMetricCard` нужно обновить отдельно. Митигация: визуальные примитивы вынесены в `src/lib/util.js` и `src/styles.css` как переиспользуемые токены.
- **Уже виден дрифт.** Пользователь зафиксировал в Sprint 2: «pinned-карточка немного отличается визуально от Catalog drawer — не критично, оставим». Это tech debt для unified design system (Sprint 3+).
- **Дублирование.** Часть JSX повторяется (status badge, value, delta). Принято — стоимость дублирования меньше, чем стоимость условной логики.

**Последствия:**

- Будущие compact-виды карточек метрик (Saved list, Home dashboard, hover-card в Graph) — **новые компоненты**, не флаги у `MetricPreview`.
- Tech debt в JTBD 9b: «pinned-карточка vs Catalog drawer cosmetic difference» решается как **отдельный шаг** (унификация дизайн-системы), не возвращением к варианту A.
- При будущем рефакторинге дизайн-системы — выделять примитивы (status badge, sparkline, value formatter) как shared component'ы, а не объединять контекстные карточки в один большой компонент с флагами.

**Что НЕ решено этим ADR:**

- Нужна ли отдельная storybook-подобная галерея для compact-видов? Open question, решаем когда compact-видов станет 3+.
- Стоит ли вынести status-badge / sparkline / value-formatter в `src/components/primitives/`? Кандидат на Sprint 3+ при росте числа потребителей.

## ADR-015: Filter-driven graph — стратегия скейла Graph view на 236+ метрик

**Дата:** 2026-04-27
**Статус:** принято
**Контекст возникновения:** Sprint 2 — переход с 47 захардкоженных метрик на 236 из seed JSON

**Контекст:**

Прототип Graph view от Claude Design был спроектирован под 47 метрик. При загрузке 236 реальных метрик из seed JSON стало видно: layout перегружен, ноды наезжают друг на друга, читаемость падает, investigation use case ломается. Это вскрылось только когда поменяли источник данных — UX-валидация на маленьком наборе оказалась нерепрезентативной.

Рассматривались альтернативы для скейла:

1. **Force-directed layout** (D3 / vis.js / Cytoscape) — авто-раскладка узлов на основе физической симуляции. Стандартная индустриальная практика для графов 100-1000 узлов.
2. **Clustering** — автогруппировка похожих метрик в кластеры с возможностью разворачивания. Например, по категории (Acquisition / Retention / Revenue / Risk).
3. **Virtualization** — виртуальный скроллинг внутри SVG, рендер только видимой области графа.
4. **Сворачивание отдельных нод** — пользователь руками скрывает неинтересные ноды.
5. **Filter-driven graph** — граф показывает только то, что прошло через FiltersDrawer. Без фильтра — дефолтное представление (возможно, только L1 + важные L2). Фильтр = drill-down.

**Решение:**

**Filter-driven graph.** Graph view — не overview map всего каталога, а **investigation tool**: «отфильтруй до релевантного среза → исследуй связи внутри среза». FiltersDrawer (тот же что в Catalog) определяет видимый набор нод. По умолчанию граф показывает компактный набор (L1 + связанные L2), пользователь сужает фильтрами до интересующего скоупа.

**Аргументы за (filter-driven):**

- **Соответствует продуктовому use case.** Аналитик в Graph не «смотрит на всю компанию» — он **расследует**: «что связано с FTD? покажи только Retention метрики. покажи только метрики с alerts.» Граф = drill-down, а не overview.
- **Консистентность с Catalog.** Тот же FiltersDrawer, тот же набор фильтров (Alerts → Level → Priority → Owner → Freq → Category). Пользователь не учится двум разным фильтр-системам.
- **State-простота.** Один источник истины — `filter` Set'ы. Не нужны отдельные мета-состояния (collapsed clusters, hidden nodes, layout positions). Предсказуемое поведение.
- **Безопасный путь.** Если filter-driven окажется недостаточно — поверх него всегда можно добавить force-directed или clustering. Обратное — сложнее (force-directed из MVP в filter-driven — большой откат).
- **Не требует внешних либ.** D3 / Cytoscape добавляют 100-300KB в bundle и сложную модель state'а. Текущий manual layout (по уровням L1/L2/L3) держится на простом коде, проще менять.

**Почему отклонены альтернативы:**

| Альтернатива | Почему отклонено |
|---|---|
| **Force-directed (D3 / Cytoscape)** | Авто-раскладка визуально красива, но непредсказуема для пользователя — каждое открытие графа node'ы в новых местах, привыкания к расположению нет. Плюс +100-300KB bundle и сложная модель state. |
| **Clustering (auto-groups)** | Группировка по категории + expand требует UX-логики (когда раскрывать, как сворачивать обратно, что с pinned внутри свёрнутого кластера). Усложнение state'а ради того, что фильтры решают чище. |
| **Virtualization** | Решает только performance, не readability. 236 нод в SVG не дают performance-проблемы — проблема визуальной перегрузки, не FPS. |
| **Сворачивание отдельных нод** | State hell: отдельный Set свёрнутых нод, конфликт с pin (что если нода свёрнута, но pinned?), undoable history. Обсуждалось и отклонено в Sprint 2 fix-cycle 1. |

**Аргументы против / трейд-оффы:**

- **Пользователь не видит «всё разом».** Если он не знает что искать — нужно сначала придумать фильтр. Митигация: дефолт показывает L1 (стратегические метрики) + связанные L2, что даёт обзорную картинку без действий.
- **Узнаваемость графа меньше.** Tableau-style force-directed выглядят более «BI-like». Но ODIN не Tableau — позиционируется как investigation tool, не dashboard.
- **Связи между метриками за пределами фильтра не видны.** Если FTD связан с Bonus Cost, но Bonus Cost отфильтрован — пользователь не увидит связь. Митигация: при hover/pin на ноде её прямые соседи подсвечиваются (даже если они «скрыты» — показываются как «грей-out» или label-only). Деталь реализации — Sprint 3.

**Последствия:**

- **Sprint 3 берёт scope:** реализация filter-driven graph. Текущий manual layout по L1/L2/L3 остаётся, но число видимых нод управляется фильтрами.
- **JTBD 9b: «Graph scaling на 236 нодах»** перестаёт быть открытой архитектурной проблемой — план есть.
- **Не подключаем D3 / Cytoscape / vis.js** в основной код прототипа. Если в будущем понадобится сложная raskладка — это отдельный архитектурный пересмотр.
- **Документ архитектуры (`architecture.md`)** должен зафиксировать что Graph view = filter-driven investigation tool, не overview map. Часть позиционирования.

**Что НЕ решено этим ADR:**

- Конкретный дефолтный набор нод при пустых фильтрах (только L1? L1 + Alerts? L1 + L2 связанные с alerts?). Финализируем в Sprint 3 при реализации.
- Поведение «связь к скрытой ноде» при hover/pin (показывать как грей-out / label-only / скрывать). Финализируем при реализации.

## ADR-016: Двух-этапный MVP — read-only Catalog (MVP-1) → full ODIN (MVP-2)

**Дата:** 2026-04-27
**Статус:** принято
**Заменяет / уточняет:** ADR-011 (WYSIWYG обязателен для MVP-2, не для MVP-1)

**Контекст:**

`roadmap.md` описывает «целостный MVP» (M1-M9) включающий M-EDIT (WYSIWYG-редактор), M-CRED (credentials store), M-SYNC (git remote sync), M7-M8 (Хугин + LLM). Это большой кусок работы — реалистично 3-4 месяца вечерами при текущем темпе.

В ходе Sprint 2 (переход на 236 реальных метрик, унификация UI) стало понятно: уже сейчас можно собрать **самостоятельный публикуемый артефакт** — read-only Catalog с реальными iGaming-метриками, Graph view, drawer спецификацией. Без редактора, без LLM, без sync — но рабочий и показательный.

Рассматривались два подхода:

1. **Целостный MVP (текущий roadmap).** Не публикуем ничего до M9. Всё или ничего.
2. **Двух-этапный MVP.** MVP-1 — read-only Catalog как отдельная промежуточная веха (и потенциальная публикация). MVP-2 — полный продукт с WYSIWYG/sync/LLM.

**Решение:**

**Двух-этапный MVP:**

### MVP-1 — Read-only Metric Catalog (~1 месяц при текущем темпе)

**Скоуп:**
- Catalog view с фильтрами (Alerts / Level / Priority / Owner / Freq / Category)
- Graph view (filter-driven, см. ADR-015) с multi-pin
- Drawer метрики (full spec) — read-only
- 236 метрик из seed JSON (уровень — синтетика для прототипа, M4 заменит на DuckDB+Parquet)
- Базовая Electron-обёртка (M3) — опционально, может быть и web-prototype для MVP-1
- **Без редактора, без credentials store, без sync, без Хугина, без LLM**

**Чем самостоятелен:**
- Можно запустить локально, посмотреть, отфильтровать, исследовать связи
- Демонстрирует фундамент: семантика метрик, разделение L1/L2/L3, investigation use case
- Открытый вопрос — публиковать ли как public artifact (GitHub Pages read-only / Electron alpha). Решается ближе к финалу MVP-1.

**Целевой milestone-набор:** M1 (закрытие UX-долгов) + M2 (миграция в MD) + опционально M3 (Electron) + M4 (DuckDB + синтетика, частично).

### MVP-2 — Full ODIN (~3-4 месяца после MVP-1)

**Скоуп:**
- M-CRED — credentials store
- M-SYNC — git remote sync (Этап 1 из ADR-013)
- M-EDIT — WYSIWYG-редактор + history UI (см. ADR-011)
- M5 — Munin Sync Contract + Parquet ingestion
- M6 — anomaly-заметки (контент)
- M7-M8 — Хугин + LLM integration (Ollama / Claude Desktop)
- M9 — demo-видео, README, релиз

**Чем самостоятелен:**
- Полноценный privacy-first BI tool для аналитиков
- Demo-сценарий 3 (Хугин расследование) работает end-to-end
- Можно отдавать пилотным пользователям

**Аргументы за:**

- **Time-to-first-publish.** За ~1 месяц получаем артефакт, который можно показать (даже если не публиковать публично — достаточно для проверки концепции на знакомых аналитиках). vs 3-4 месяца до любой demonstrable вехи.
- **Реалистичная оценка с обновлённым темпом.** Sprint 2 + два fix-cycle закрыты за один продуктивный сеанс. Темп выше прогноза, MVP-1 в ~1 месяц — реалистично.
- **Валидация концепции до больших инвестиций.** Если MVP-1 показывает что аналитики не понимают семантический слой / не видят ценности — лучше узнать это до того как 2-3 месяца тратить на M-EDIT.
- **Психологическое.** Один большой артефакт через 6 месяцев — высокий риск выгорания. Промежуточная веха через месяц — поддерживает мотивацию и подтверждает прогресс.
- **Снижает риск scope creep.** MVP-1 имеет жёсткую границу: «никакого редактора, никакого LLM». Разработчик (Code) и PM (Cowork) не пытаются впихнуть «ещё одну фичу» под видом MVP.
- **Не противоречит ADR-011.** ADR-011 утверждает что **WYSIWYG обязателен в MVP** — да, в **MVP-2**. MVP-1 — не для аналитиков-редакторов, а для аналитиков-читателей и для self-validation концепции.

**Аргументы против / трейд-оффы:**

- **Соблазн застрять на MVP-1.** Психологически легче полировать read-only артефакт бесконечно, чем приступать к большому M-EDIT. Митигация: после CLOSE MVP-1 — обязательная phase «MVP-2 kickoff» с переоценкой скоупа.
- **MVP-1 не решает заявленную боль ICP.** Аналитики без редактора получают «красивый каталог» — не «инструмент для команды». Это тренировочный артефакт, не финальный продукт. Нельзя позиционировать MVP-1 как «решение для iGaming data team».
- **Двойная работа над публикацией.** Если MVP-1 публикуется отдельно — нужен README, демо-описание, скриншоты дважды (для MVP-1 и MVP-2). Митигация: для MVP-1 публикация **опциональна** — можно использовать только для self-validation, не показывать миру.
- **Риск раздробления focus.** «Что в MVP-1, что в MVP-2» — постоянно перетягиваемая граница. Митигация: явные acceptance criteria в roadmap.md и JTBD-full-scope.md, фиксируется при CLOSE Sprint 2.

**Последствия:**

- **`roadmap.md` обновляется** — добавляется маркер `MVP-1 boundary` между M2/M4 и M-CRED/M-EDIT/M5+. Внутренние milestones (M1-M9) сохраняются как есть, добавляется группировка.
- **ADR-011 уточняется** — статус «WYSIWYG обязателен в MVP» означает в **MVP-2**. MVP-1 остаётся read-only, ADR-011 не нарушается потому что MVP-1 не позиционируется как «инструмент для аналитиков-редакторов» — он внутренний/тестовый артефакт.
- **JTBD-full-scope.md** получает разметку user stories по MVP-1 / MVP-2 (M1-M2-частично-M4 vs M-CRED-M-SYNC-M-EDIT-M5-M6-M7-M8-M9).
- **Sprint planning меняется.** Sprint 3, 4, возможно 5 — фокус на MVP-1 (Graph polish, Catalog scroll, frontmatter ADR, vertical slice MD). После CLOSE MVP-1 — переоценка скоупа MVP-2.
- **Публикация MVP-1 — open question.** Может быть приватный share с двумя-тремя аналитиками для фидбэка, либо публичная read-only демка через GitHub Pages (как уже упоминается в roadmap.md строка 173). Решаем ближе к CLOSE MVP-1.

**Что НЕ меняется:**

- Финальный MVP всё ещё MVP-2 — privacy-first BI с WYSIWYG, sync, LLM. Двух-этапность не отменяет амбицию, добавляет промежуточную точку.
- Все ADR (007-013) остаются в силе. MVP-1 не нарушает ни один из них (он подмножество).
- ADR-008 (local-first), ADR-010 (push-based), ADR-011 (WYSIWYG) — все валидны для MVP-2.

**Открытые вопросы (откладываются):**

- Публиковать ли MVP-1 публично или приватный share-only? Решаем при CLOSE MVP-1.
- Должен ли MVP-1 быть Electron-app или достаточно web-prototype (как сейчас)? Решаем когда подойдём к M3.
- Сколько спринтов уйдёт на MVP-1 из текущей точки? Прикидка 3-5 спринтов. Финализируем при формировании Sprint 3.
