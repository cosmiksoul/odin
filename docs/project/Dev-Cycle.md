# Dev Cycle для ODIN

> Формализованный цикл разработки проекта. Адаптация процесса из Ratanote под специфику ODIN (desktop Electron, несколько типов артефактов, длинные milestones).

> Последнее обновление: 2026-04-25

---

## Участники и роли

| Роль | Кто | Что делает |
|------|-----|-----------|
| Product Owner / Designer / QA | Пользователь (N13Z) | Принимает все продуктовые и архитектурные решения. Тестирует на машине. |
| PM / Architecture / Documentation | Claude Cowork | Планирование спринтов, формирование ADR совместно с пользователем, поддержка всей документации, sprint prompts, code review, тест-кейсы |
| Developer | Claude Code | Пишет код, делает sprint-reports, исправляет баги |

**Принципы:**
- Cowork и Code — **разные инстансы с разным контекстом**. Cowork не трогает код. Code не занимается планированием и архитектурой.
- Cowork **может редактировать** все документы в `docs/` и `CLAUDE.md`. Правки в `docs/context/` (ADR, концепция, архитектура) — только по результатам обсуждения с пользователем, не самостоятельно.
- Пользователь — единственный источник продуктовых решений и единственный, кто видит реальный результат на устройстве.

## Типы спринтов

В ODIN не все спринты одинаковы. Разные типы требуют разных циклов:

### A. Code sprint (стандартный)

Спринт по разработке фичи или фикса. Полный 8-фазный цикл (см. ниже). Большинство спринтов — этого типа.

**Примеры:** "M3-1: setup Electron-обёртки", "M-EDIT-2: TipTap редактор для body", "M-SYNC-1: UI подключения git remote".

### B. Content sprint

Спринт без кода — **создание контента**. Урезанный цикл: только PLAN → CONTENT → REVIEW → CLOSE. Нет phase TEST PREP / QA / FIX / RETEST в традиционном виде.

**Примеры:**
- M6: написание 3-5 anomaly-заметок (текст с реальной аналитической логикой)
- Подготовка documentation для пользователей (installation guide, git remote setup guide)
- Обновление seed data — добавление новых метрик в `default-vault/_seed/igaming_metrics.json`

**Особенности:**
- "Тестирование" = ревью смысла. Пользователь читает, оценивает: понятно? полезно? правдоподобно?
- Нет sprint-report-N.md в виде "что построено / какие файлы созданы". Вместо него — короткий `content-N-review.md` с заметками автора.
- Закрытие на phase CLOSE — обновление JTBD (отметить созданный контент) и Timeline в CONTEXT.

### C. Architecture sprint

Спринт с **архитектурной работой до начала кода**. Нужен когда задача требует решений по технологиям/паттернам, которых ещё нет в ADR.

**Примеры:**
- M-CRED: дизайн интеграции keytar (где хранить service name, как обращаться, fallback на Linux)
- M-SYNC: дизайн потоков push/pull, обработка credentials, UX-сообщений при конфликтах
- Финализация frontmatter-схемы перед M-EDIT

**Особенности цикла:**
- Phase PLAN расширена: включает **research + проектирование** (живёт в `docs/project/sprint-N-design.md` — рабочий документ под Architecture sprint)
- Перед DEV phase — **review архитектурного решения** пользователем
- Если решение значимое и долгоживущее — Cowork в диалоге с пользователем формирует новый ADR в `docs/context/decisions-log.md`
- Дальше — обычный Code sprint по реализации согласованного дизайна

## Связь milestones и спринтов

Из roadmap (M1-M9 + M-CRED, M-SYNC, M-EDIT) milestones — это **виртуальные метки**, а не дисциплинирующие границы.

**Один milestone = N спринтов внутри.**

Пример:
- M-EDIT (in-app WYSIWYG-редактор + history UI) — это **3-5 спринтов**:
  - M-EDIT-1: research + выбор библиотеки (TipTap vs Lexical vs ProseMirror) → **Architecture sprint**
  - M-EDIT-2: базовый редактор для body → **Code sprint**
  - M-EDIT-3: формочки для frontmatter → **Code sprint**
  - M-EDIT-4: history UI (просмотр изменений) → **Code sprint**
  - M-EDIT-5: rollback + diff renderer → **Code sprint**

Cowork при планировании каждого следующего спринта смотрит **что фактически сделано в JTBD-full-scope.md**, а не "пора по плану закрывать M-EDIT".

**Правило:** не привязываться жёстко к "1 milestone = 1 спринт". Брать в спринт то, что осмысленно сделать за 2-5 дней работы вечерами.

## Цикл Code sprint (стандартный)

Каждый Code sprint проходит через 9 фаз. Фазы 6-8 могут повторяться (цикл фиксов), но это **штатная часть процесса**, а не аварийный спринт.

```
  ┌──────────────────────────────────────────────────────────────────────┐
  │                          SPRINT N                                    │
  │                                                                      │
  │  1. PLAN ──► 2. PROMPT ──► 3. DEV ──► 4. CODE REVIEW ──► 5. TEST PREP│
  │                                                              │       │
  │                                                              ▼       │
  │  9. CLOSE ◄── 8. RETEST ◄── 7. FIX ◄────────────────── 6. QA        │
  │       │            │                                                 │
  │       │            └─── (если есть новые баги) ────► 7. FIX          │
  │       │                                                              │
  │       └──► SPRINT N+1                                                │
  └──────────────────────────────────────────────────────────────────────┘
```

**Краткая суть фаз:**
- **PLAN, PROMPT** — Cowork готовит задачу
- **DEV** — Code пишет
- **CODE REVIEW** — Cowork проверяет код против ADR/scope/architecture, формирует список concerns
- **TEST PREP** — Cowork готовит тест-кейсы (с учётом результатов code review)
- **QA** — пользователь тестирует на машине
- **FIX, RETEST** — устранение багов (циклически, до 2 итераций)
- **CLOSE** — обновление документации, закрытие спринта

### Фаза 1: PLAN (Cowork + Пользователь)

**Цель:** определить что делаем в спринте и зафиксировать scope.

**Действия:**
- Обсудить текущее состояние проекта (что готово, что блокирует) — Cowork читает последние sprint-reports и JTBD
- Выбрать scope из `docs/project/JTBD-full-scope.md`
- Определить цель спринта одним предложением
- **Проверить scope против ADR** — не противоречит ли цель архитектурным решениям? Если есть конфликт — стоп, обсуждаем с пользователем (возможно нужен новый ADR)
- Обсудить риски и неясности до начала разработки
- Если есть UX-решения — принять их **до** передачи в разработку

**Артефакт:** обсуждение в Cowork. Финальный scope фиксируется в sprint prompt (Фаза 2).

**Правила:**
- Scope должен быть реалистичным. Лучше сделать меньше и закрыть, чем растянуть спринт.
- Если в процессе обсуждения всплывают вопросы, требующие исследования — это **Architecture sprint**, не часть планирования Code sprint.
- Пользователь имеет право вето на любую фичу в scope.

**Анти-паттерн (из истории проекта):** Cowork **не должен** сам решать "это можно отложить, для демо не критично" — это путь к ошибкам типа отозванного ADR-009. Если есть подозрение, что фича должна быть в скоупе но "сложная" — обсуждать **до** урезания, не после.

---

### Фаза 2: PROMPT (Cowork)

**Цель:** подготовить детальное задание для Claude Code.

**Действия:**
- Написать `sprint-N-prompt.md` — полное ТЗ для Claude Code
- Включить: цель, список задач, технические детали, ограничения (DO NOT), критерии приёмки
- Сослаться на нужные файлы (`CLAUDE.md`, предыдущие отчёты, релевантные ADR)

**Артефакт:** `docs/project/sprint-N-prompt.md`

**Шаблон prompt-файла:**

```markdown
# Sprint N — [Название]

## Overview
[Цель спринта одним абзацем]

## Tasks
[Детальный список задач с техническими деталями]

## Technical Notes
[Архитектурные решения, зависимости, ссылки на документы]

## ADR Constraints
[Какие ADR применимы к этому спринту, что нельзя нарушать]

## Testing
[Что проверить после реализации, smoke-test]

## DO NOT
[Что НЕ трогать, ограничения]
```

**Правила:**
- Prompt должен быть **самодостаточным** — Claude Code не читает историю чатов из Cowork
- Чем детальнее prompt, тем лучше результат. Не жалеть времени на эту фазу.
- Если есть UX-решение — описать его в prompt или сослаться на файл/screenshot
- **Обязательно** упомянуть какие ADR применимы (ссылки на `docs/context/decisions-log.md`)

---

### Фаза 3: DEV (Claude Code)

**Цель:** реализация задач спринта.

**Действия:**
- Пользователь запускает Claude Code с prompt-файлом
- Claude Code пишет код, делает коммиты
- По завершении формирует отчёт

**Артефакт:** `docs/project/sprint-report-N.md`

**Шаблон отчёта** (Claude Code должен следовать этому формату):

```markdown
# Sprint N Report — [Название]

**Dates**: YYYY-MM-DD
**Status**: Complete / Partial

## Goal
[Цель спринта]

## What was built
[Описание реализованного с деталями]

## Files Created
| File | Purpose |
|------|---------|

## Files Modified
| File | Changes |
|------|---------|

## ADR Compliance
[Какие ADR были применимы и как они соблюдены. Если что-то нарушено — обоснование почему.]

## Known Issues
[Что не удалось, ограничения, технический долг]

## Notes
[Дополнительные наблюдения]
```

**Правила:**
- Пользователь может задавать вопросы Claude Code во время разработки, но **крупные изменения scope** — через Cowork.
- Если Claude Code сталкивается с проблемой, которая меняет scope — **остановиться**, вернуться к обсуждению в Cowork.
- Если Claude Code сталкивается с противоречием ADR — **остановиться**, вернуться к Cowork и пользователю для обсуждения. Не пытаться "обойти" ADR молча.

### Фаза 4: CODE REVIEW (Cowork)

**Цель:** независимая проверка кода другим LLM-инстансом до того как пользователь начнёт QA.

**Зачем:** Code и Cowork — разные инстансы с **разным контекстом**. Cowork знает все ADR, видит sprint prompt, читал прошлые reports. Code писал код фокусируясь на задаче. Cowork ловит то что Code пропустил из-за scope-tunnel-vision: нарушения ADR, скоуп-крип, несогласованность с архитектурой, нарушения anti-patterns.

**Что Cowork делает:**
- Читает `sprint-N-prompt.md`, `sprint-report-N.md`, git diff кода спринта
- Проверяет:
  - **ADR compliance** — нет ли нарушений зафиксированных решений (cloud-зависимости, вход через git CLI вместо WYSIWYG, обход credentials store, etc)
  - **Scope** — каждое изменение трассируется к sprint prompt? Нет ли лишних правок?
  - **Architecture** — код лёг в правильные модули (`lib/credentials/`, `lib/git/`, `lib/duckdb/`)?
  - **Anti-patterns** из CLAUDE.md — не появилось ли что-то из списка
  - **Согласованность с прошлыми спринтами** — паттерны не противоречат тому что делали раньше
- Формирует список concerns с категориями (см. ниже)

**Что Cowork НЕ делает:**
- Не предлагает альтернативный код ("я бы написал через reduce, а не через map")
- Не пишет рефакторинг сам
- Не углубляется в стиль кода (это уровень CLAUDE.md правил Карпатого, проверяет Code сам)
- Не запускает код (это юзер на QA)

**Артефакт:** `docs/project/code-review-sprint-N.md`

**Шаблон:**

```markdown
# Code Review Sprint N — [Название]

**Reviewer:** Cowork
**Date:** YYYY-MM-DD

## Summary
[1-2 предложения общей оценки]

## Concerns

### 🔴 Blockers (требуют fix до QA)
| # | Где | Что не так | Какой ADR/правило |
|---|-----|-----------|-------------------|
| 1 | `src/lib/llm/anthropic.ts:42` | Anthropic API key читается из process.env, а не keytar | ADR-012 |

### 🟡 Concerns (обсуждаемые)
| # | Где | В чём concern |
|---|-----|---------------|
| 1 | `src/components/MetricCard.tsx` | 200+ строк в одном компоненте, может стоит разбить |

### 🟢 Notes (наблюдения, не требуют фикса в этом спринте)
| # | Где | Заметка |
|---|-----|---------|
| 1 | `src/lib/git/` | В будущем стоит добавить retry логику для push'а — сейчас не критично |

## ADR Compliance Check
- ADR-008 (local-first): ✅ нет cloud-зависимостей в diff
- ADR-011 (WYSIWYG): ✅ нет упоминаний git/MD в UX-строках
- ADR-012 (credentials): 🔴 см. Blocker #1

## Scope Compliance
✅ Все изменения трассируются к задачам в sprint prompt
[или: 🟡 файл X изменён — это в скоупе?]

## Decision Log
[Заполняется после диалога юзер ↔ Cowork по concerns]
```

**Правила:**
- Cowork **не угадывает** — если что-то не уверен, ставит 🟡 Concern с вопросом, не 🔴 Blocker
- 🔴 Blocker = объективное нарушение зафиксированного правила (ADR, anti-pattern из CLAUDE.md)
- 🟡 Concern = "подозрительно, обсуди"
- 🟢 Note = "на будущее, не сейчас"

**Дальше — диалог юзер ↔ Cowork:**

По каждому concern юзер принимает решение:
- **"Согласен — fix"** → переходит в backlog для FIX phase (как обычные баги)
- **"Не согласен — discard"** → отмечается в Decision Log с обоснованием
- **"Нужно обсудить"** → разговор, итог фиксируется

Спорные моменты могут привести к одному из:

| Что обсудили | Куда фиксировать |
|---|---|
| Архитектурное решение длительного действия | Новый ADR в `docs/context/decisions-log.md` (Cowork формирует по итогам диалога с пользователем) |
| Стандарт кода для всего проекта | CLAUDE.md, секция `## ODIN-specific правила` |
| Project-specific (только этот sprint) | `code-review-sprint-N.md` секция Decision Log |
| Замечание не сделали, но оно общее | `docs/project/CONTEXT.md` (Timeline / Tech debt) |

**Важно:** правки в `docs/context/` (ADR, концепция, архитектура) и в `CLAUDE.md` Cowork делает **только по результатам обсуждения с пользователем**, не самостоятельно. Если есть сомнение — спрашивает.

**Анти-паттерны на этой фазе:**
- ❌ Cowork и Code начинают переписывать друг друга в кругу — модерируешь ты
- ❌ Cowork предлагает рефакторинг "потому что я бы сделал иначе" — это не его scope
- ❌ Юзер пропускает Blockers потому что "потом починю" — Blocker блокирует QA до фикса

**Когда фаза заканчивается:**
- Все 🔴 Blockers разрешены (либо fix-нуты, либо переквалифицированы в 🟡/🟢 после обсуждения)
- 🟡 Concerns обсуждены, по каждому есть решение в Decision Log
- 🟢 Notes зафиксированы (если применимо — в tech-debt или CONTEXT)

### Фаза 5: TEST PREP (Cowork)

**Цель:** подготовить тест-кейсы для приёмки.

**Действия:**
- На основе sprint-report, sprint-prompt и code-review результатов создать `test-cases-sprint-N.md`
- Включить: позитивные сценарии, негативные сценарии, edge cases, регрессию предыдущих спринтов
- **Особо проверить участки, где были 🟡 Concerns** в code review — там высокая вероятность найти runtime-баги
- Каждый тест-кейс: описание, ожидаемый результат, поле для статуса

**Артефакт:** `docs/project/test-cases-sprint-N.md`

**Шаблон тест-кейса:**

```markdown
| # | Test Case | Expected | Status |
|---|-----------|----------|--------|
| 1 | [Описание действия] | [Что должно произойти] | ⬜ |
```

Статусы: `⬜` (не проверен), `ok` (пройден), `bug` (баг, описать в таблице багов ниже).

**Правила:**
- Тест-кейсы пишутся **до** начала тестирования, а не параллельно
- Регрессионные тесты обязательны (ключевые сценарии предыдущих спринтов)
- Количество тест-кейсов разумное — 15-30 на спринт, не 100

**ODIN-специфика тестов:**
- Тесты на корректность frontmatter — каждое изменение схемы проверяется автотестом + ручным
- Тесты на git-операции — commit/log/rollback с проверкой состояния файлов
- Тесты на Electron-специфику — main process / renderer / IPC
- Smoke-тест Хугина — MCP-сервер запускается, отвечает на простой запрос (для спринтов M7+)

---

### Фаза 6: QA (Пользователь)

**Цель:** протестировать на машине и зафиксировать результаты.

**Действия:**
- Собрать и запустить ODIN: `npm run dev` (для разработки) или `npm run build` + установка .dmg/.exe (для финального QA)
- Пройти по тест-кейсам, отмечая статусы
- Найденные баги — описать в таблице багов с severity
- По возможности — скриншоты или описание шагов воспроизведения

**Артефакт:** обновлённый `docs/project/test-cases-sprint-N.md` с заполненными статусами и таблицей багов.

**Severity:**
- **Critical:** приложение крашится или теряет данные
- **High:** основная фича спринта не работает
- **Medium:** фича работает, но с заметными проблемами
- **Low:** косметика, мелкие неудобства

**ODIN-специфика QA:**
- Тестирование в **обеих темах** (светлая/тёмная) для UI-фичей
- Если фича влияет на git — проверить через CLI что коммиты правильные (`git log`, `git status`)
- Для фичей с credentials — проверить что секреты в Keychain/Credential Manager, **не** в файлах vault'а

---

### Фаза 7: FIX (Cowork + Claude Code) — если есть баги

**Цель:** исправить найденные баги.

**Действия:**
- Обсуждение результатов QA в Cowork
- Классификация багов: fix now (Critical/High) vs defer (Medium/Low)
- Подготовка fix prompt: `docs/project/sprint-N-fix-prompt.md`
- Claude Code выполняет фикс
- Claude Code обновляет sprint-report (добавляет секцию "Bug Fixes")

**Артефакт:** `docs/project/sprint-N-fix-prompt.md`, обновлённый `docs/project/sprint-report-N.md`

**Правила:**
- Это **не отдельный спринт**. Это штатная фаза текущего спринта.
- Fix prompt должен содержать: ID бага, описание проблемы, предлагаемый фикс, что НЕ трогать
- Deferred баги переносятся в backlog с пометкой из какого спринта

---

### Фаза 8: RETEST (Пользователь) — если были фиксы

**Цель:** проверить что фиксы работают и ничего не сломалось.

**Действия:**
- Перепроверить тест-кейсы с багами (те, что были `bug`)
- Пройти ключевые регрессионные тесты
- Обновить статусы в test-cases

**Артефакт:** обновлённый `docs/project/test-cases-sprint-N.md`

**Правила:**
- Если после фикса появились новые баги — вернуться в Фазу 7, но **не более 2 итераций**. Если после 2 итераций остаются баги — deferred в следующий спринт.
- Финальный вердикт за пользователем: "принято" или "нужна ещё итерация".

---

### Фаза 9: CLOSE (Cowork)

**Цель:** закрыть спринт и обновить документацию.

**Действия:**
- **Обновить `docs/project/JTBD-full-scope.md`:** отметить `[x]` завершённые user stories, проставить теги ★ если применимо
- **Обновить `CLAUDE.md`** (если изменилось — например добавлены новые anti-patterns)
- **Обновить `docs/project/CONTEXT.md`:** добавить запись в Development Timeline (что закрыто, какие milestones продвинулись)
- Deferred баги — зафиксировать (куда перенесены)
- Краткое ретро: что сработало, что нет, что улучшить в процессе

**Артефакты:** обновлённые `JTBD-full-scope.md`, `CLAUDE.md`, `CONTEXT.md`.

**Правила:**
- Спринт **не закрывается** пока документация не обновлена. Это обязательный шаг.
- Ретро не обязано быть длинным — 3-5 предложений достаточно.
- Если в спринте всплыли архитектурные вопросы — обсудить с пользователем, по итогу Cowork формирует новый ADR в `decisions-log.md`.

## Цикл Content sprint (урезанный)

Для спринтов без кода (M6 anomaly-заметки, документация, обновление seed data) — упрощённый процесс:

### Фаза 1: PLAN
То же что Code sprint — выбор скоупа, цель.

### Фаза 2: CONTENT (Cowork или Пользователь)
Создание контента. Может быть совместно — Cowork делает черновик, пользователь дорабатывает.

**Артефакт:** контент-файлы (anomaly-заметки в `default-vault/metrics/<id>/anomaly-*.md`, документация в `docs/user/`, обновлённый seed JSON).

### Фаза 3: REVIEW (Пользователь + опционально Cowork)
Пользователь читает результат, оценивает:
- Понятно ли?
- Реалистично ли (для anomaly-заметок — отражает ли реальную аналитическую логику)?
- Соответствует ли формату (frontmatter, wiki-links)?

**Артефакт:** короткий `content-N-review.md` с заметками "что хорошо, что переписать".

### Фаза 4: CLOSE
Обновление JTBD и CONTEXT, как в Code sprint.

**Особенность Content sprints:** контент часто требует **2-3 итераций**. Не "написал → принял", а "черновик → ревью → правки → финал".

## Цикл Architecture sprint (с research-фазой)

Для спринтов, где требуется архитектурная работа до кода:

### Фаза 1: PLAN
То же. Цель = "решить архитектурный вопрос X и подготовить scope для следующего Code sprint".

### Фаза 2: RESEARCH + DESIGN (Cowork)
- Изучение вариантов (через web search, изучение docs библиотек)
- Подготовка `sprint-N-design.md` с обоснованием выбора
- Возможные альтернативы с trade-off'ами
- Рекомендация

**Артефакт:** `docs/project/sprint-N-design.md`

### Фаза 3: REVIEW (Пользователь + Cowork)
- Пользователь оценивает рекомендацию
- Если решение значимое и долгоживущее — Cowork по итогам диалога формирует новый ADR в `docs/context/decisions-log.md`

**Артефакт:** утверждённый design doc + опционально новый ADR.

### Фаза 4: HAND-OFF
Architecture sprint **завершается** передачей в следующий Code sprint. design doc становится частью sprint prompt'а следующего спринта.

**Особенность:** Architecture sprint **не пишет код**. Если в его рамках написан какой-то PoC — это отмечается отдельно как "experimental, не идёт в main".

## Правила для документации

1. **`docs/project/JTBD-full-scope.md`** — обновляется на фазе CLOSE каждого спринта. Чек-боксы отражают реальное состояние.
2. **`CLAUDE.md`** — источник правды для Claude Code. Обновляется когда появляются новые anti-patterns или изменяются ADR.
3. **`docs/project/CONTEXT.md`** — история проекта. Каждый закрытый спринт — новая запись в Timeline.
4. **`docs/context/`** — стабильные документы (концепция, архитектура, ADR). Cowork **может редактировать**, но только по результатам обсуждения с пользователем. Самостоятельные правки — нет.
5. **Sprint report** — пишется Claude Code, формат стандартизирован (шаблон выше).
6. **Test cases** — пишутся Cowork **до** тестирования. Заполняются пользователем **после**.

## Чеклист смены фаз (Code sprint)

Переход из фазы в фазу — через явное подтверждение в Cowork.

| Переход | Что нужно проверить |
|---------|-------------------|
| PLAN → PROMPT | Scope согласован, UX-вопросы закрыты, ADR-конфликтов нет |
| PROMPT → DEV | Prompt файл создан и просмотрен пользователем |
| DEV → CODE REVIEW | Sprint report получен от Claude Code |
| CODE REVIEW → TEST PREP | Все 🔴 Blockers разрешены, 🟡 Concerns обсуждены, Decision Log заполнен |
| TEST PREP → QA | Test cases файл создан |
| QA → FIX | Test cases заполнены, баги классифицированы |
| FIX → RETEST | Fix report получен, код обновлён |
| RETEST → CLOSE | Все Critical/High баги закрыты или осознанно deferred |
| CLOSE → следующий PLAN | Документация обновлена (JTBD, CONTEXT, при необходимости CLAUDE.md) |

## Калибровка Cowork — типичные ошибки и как их избегать

> Это раздел извлечён из истории работы над ODIN. Cowork должен **проверять самого себя** на эти паттерны.

### Ошибка 1: Подмена продуктовой логики демо-логикой

**Симптом:** "Эта фича не нужна для demo-сценария → отложим из MVP".

**Почему ошибка:** demo показывает продукт в идеальных условиях. **Продукт должен работать в реальных условиях с реальной аудиторией.** Если автор называет продуктовую боль из опыта — она не отменяется аргументом "для демо не критично".

**Как избегать:** при выборе scope MVP приоритет = что нужно пользователю, а не что красивее в видео.

**Конкретный кейс:** ADR-009 был отозван из-за этой ошибки. Изначально я (Cowork) предложил исключить in-app редактор из MVP, потому что он "не нужен для Хугин-демо". Это противоречило прямому продуктовому требованию автора (аналитики не используют git). Решение пересмотрено в ADR-011.

### Ошибка 2: Преждевременное схлопывание архитектурных решений

**Симптом:** "Запретить класс X" (no cloud, no editor, no auth) без разложения на под-категории.

**Почему ошибка:** широкий запрет съедает кейсы которых не подумали. ADR-008 изначально запрещал OAuth целиком — пришлось править, потому что это съедало credentials store, git remote auth, RBAC внутри ODIN.

**Как избегать:** когда видишь решение "запретить класс вещей", разложи класс на под-категории и явно скажи что запрещено, что — нет.

### Ошибка 3: Скоуп-крип в фазе DEV

**Симптом:** Claude Code "заодно отрефакторит соседний модуль", "улучшит соседний UI", "добавит ещё одну мелочь".

**Почему ошибка:** прямо нарушает Правило 3 из CLAUDE.md (Surgical Changes). Каждое изменение должно трассироваться к sprint prompt.

**Как избегать:** sprint prompt должен явно содержать секцию "DO NOT" с перечислением того, что **рядом**, но **не трогаем**.

### Ошибка 4: Пропуск ADR при планировании

**Симптом:** scope спринта противоречит ADR, и Cowork не заметил при PLAN.

**Почему ошибка:** Claude Code будет либо нарушать ADR, либо тратить время на эскалацию.

**Как избегать:** в фазе PLAN **обязательно** проверить — не противоречит ли цель спринта какому-то ADR. Если противоречит — это либо ошибка планирования, либо повод для нового ADR (обсуждается с пользователем, фиксируется Cowork).

## Изменения относительно Ratanote-версии

Этот документ — адаптация из Ratanote (другой проект). Что изменено:

1. **Платформа:** мобильная Android-разработка → desktop Electron
2. **QA:** APK на телефоне → `npm run dev` или .dmg на машине
3. **Добавлены типы спринтов:** Content sprint и Architecture sprint, помимо стандартного Code sprint
4. **Связь milestones и спринтов:** один milestone = N спринтов, milestones — виртуальные метки
5. **Калибровка Cowork:** новый раздел с разбором типичных ошибок (на основе истории работы над ODIN)
6. **ADR-проверка:** в фазе PLAN явная проверка на конфликт со ADR; в sprint-report секция "ADR Compliance"
7. **Разделение зон ответственности:** Cowork работает в `docs/project/`, не редактирует `docs/context/` (стабильные документы)
8. **Новая фаза CODE REVIEW** между DEV и TEST PREP — независимая проверка Cowork: ADR compliance, scope, architecture, anti-patterns. Scope ограничен flag concerns, без альтернативных реализаций. Артефакт `code-review-sprint-N.md`.
