# ODIN 2.0 — дизайн реанимации

**Дата:** 2026-06-11
**Статус:** утверждён владельцем (обсуждение в сессии Claude Code)
**Заменяет:** roadmap.md, architecture.md и большинство ADR апреля 2026 (детали в §10)

---

## 1. Цель

**Портфолио-демка с твистом: булетпруф-система «принёс parquet + схему → получил каталог метрик».**

ODIN перестаёт быть iGaming-продуктом и становится универсальным локальным каталогом метрик:
пользователь приносит Parquet-витрину с агрегатами в строгом контракте и `catalog.json` со схемой
метрик — и получает готовую визуализацию: каталог, граф зависимостей, динамику, разрезы, алерты,
MD-контекст. iGaming-сид из 236 метрик становится дефолтным демо-датасетом, а не продуктом.

Критерий успеха: незнакомый человек скачивает приложение, открывает demo-vault, видит работающий
каталог на реальных (синтетических) данных; затем подменяет vault своим parquet+схемой — и всё
работает или валидатор внятно объясняет, что не так.

### Ключевые решения (зафиксированы с владельцем)

| Вопрос | Решение |
|---|---|
| Цель | Портфолио/демка + универсальный движок («любой parquet → каталог») |
| Платформа | Десктоп: **Electron** (web-демо как второй таргет того же кода) |
| Контракт данных | **Строгий**: фиксированный long-format, никакого маппинга колонок и автодетекта |
| Схема каталога | **JSON-манифест** (`catalog.json`) — машинная истина; **MD-файлы** — человеческий контекст |
| Скоуп MVP | Ядро вьюх + дрилл-даун по разрезам + MCP-сервер + редактор-лайт |
| Не в MVP | Git-sync из UI, WYSIWYG/history UI, credentials store, RBAC, 2D-разрезы |
| Подход к коду | **Чистый кор**: фронт переписывается на TypeScript с нуля; из прототипа берём дизайн-систему и UX-решения |

## 2. Архитектура

Монорепо (npm workspaces), TypeScript везде — UI, движок, MCP, скрипты. Python не используется
(синтетика генерируется на TS через DuckDB).

```
odin/
  packages/
    core/      # доменная логика, ноль зависимостей от React/Electron:
               #   типы, zod-схемы catalog.json и контракта, валидатор,
               #   граф зависимостей, интерфейс DataProvider
    engine/    # Node-слой данных: загрузка vault, DuckDB (@duckdb/node-api),
               #   чтение parquet по контракту, валидация данных, расчёт алертов
    ui/        # React-приложение: все вьюхи, дизайн-токены (перенос styles.css)
    desktop/   # Electron: main + preload, IPC-мост ui <-> engine, electron-builder
    demo/      # web-таргет для GitHub Pages: тот же ui + StaticProvider
               #   с предрасчитанным снапшотом синтетики
    mcp/       # MCP-сервер: отдельный процесс, импортирует engine;
               #   Claude Desktop запускает его сам (НЕ child process Electron)
  vault-demo/  # демо-vault: catalog.json + context/*.md + data/*.parquet
  scripts/     # миграция сида, генерация сценарной синтетики
  docs/
```

**Инварианты:**

- `ui` знает только интерфейс `DataProvider` из `core`. В Electron провайдер ходит через IPC
  в `engine`; в web-демо читает статический снапшот. Один UI — два таргета.
- `core` не импортирует ничего платформенного (ни React, ни Node API) — чистые типы и функции.
- `engine` — единственное место, где живут DuckDB и файловая система.
- `metric_id` — единственный идентификатор метрики везде (UI-ключи, граф, localStorage, MCP).
  Identity-по-имени из прототипа не переносится.

**Стек:** React 18 + TypeScript + Vite, Zustand, electron-vite + electron-builder,
`@duckdb/node-api`, `@modelcontextprotocol/sdk`, zod, Vitest. Секретов в MVP нет (только
локальные файлы) — ни keytar, ни safeStorage не нужны.

## 3. Vault — то, что приносит пользователь

```
my-vault/
  catalog.json        # схема каталога — машинная истина
  context/<id>.md     # опциональный человеческий контекст метрики
  data/*.parquet      # значения в строгом контракте
```

Setup-флоу: открыл приложение → указал папку vault (или создал из шаблона) → валидатор прогнал
схему и данные → каталог отрисовался. Демо-режим: приложение поставляется с `vault-demo`.

## 4. catalog.json — схема каталога

На каждую метрику:

| Поле | Тип | Обяз. | Описание |
|---|---|---|---|
| `id` | string (slug, `^[a-z0-9_]+$`) | да | Единственный идентификатор; совпадает с `metric_id` в parquet |
| `name` | string | да | Отображаемое имя |
| `category` | string | да | Произвольная категория (фасет фильтра) |
| `level` | enum L1/L2/L3 | да | Уровень иерархии (структура каталога и графа) |
| `unit` | enum: number / percent / currency / duration | да | Форматирование значений |
| `frequency` | enum: daily / weekly / monthly | да | Гранулярность (месячные лежат на 1-м числе) |
| `priority` | enum: must / should / nice | нет | Приоритет (фасет фильтра; в сиде заполнен на 100%) |
| `formula` | string | нет | Текстовая формула/определение |
| `depends_on` | string[] (id) | нет | Рёбра графа зависимостей; валидатор проверяет существование |
| `dims` | string[] (dim_key) | нет | Какие разрезы у метрики ожидаются в данных |
| `alerts` | `{red?: Threshold, yellow?: Threshold}` | нет | Структурированные пороги (см. ниже) |
| `benchmarks` | `{label: string, value: string}[]` | нет | Справочные бенчмарки |
| `tags` | string[] | нет | Доп. фасет |

`Threshold` (v1, минимум): `{ op: "lt"|"gt", value: number, basis: "value"|"wow_pct" }` —
порог либо на последнее значение, либо на неделя-к-неделе изменение в процентах. Текстовые
пороги из сида, которые не парсятся в эту модель, мигрируют в MD-контекст, не в схему.

Файл валидируется zod-схемой; из неё публикуется JSON Schema (`$schema` в шапке файла) —
автокомплит и подсветка ошибок в любом редакторе бесплатно.

MD-контекст: `context/<id>.md` — свободный Markdown (ловушки, как улучшать, история аномалий).
Файл опционален; специальной схемы у него нет. Рендерится в карточке метрики, отдаётся через MCP.

## 5. Контракт данных (v2, упрощённый munin-sync-contract)

Long-format parquet, 5 колонок:

```
metric_id : string  — id из catalog.json
date      : DATE    — день (UTC); месячные метрики на 1-е число месяца
dim_key   : string  — ключ разреза; '_total' для итогового значения
dim_value : string  — значение разреза; '_total' для итога
value     : double  — значение метрики
```

Изменения относительно контракта апреля-2026 (по итогам аудита):

- **Нет колонки `value_type`** — форматирование живёт в схеме метрики (`unit`), в данных ему
  делать нечего; колонка создавала возможность противоречий.
- **Тоталы явные** (`_total`/`_total`), NULL-семантика запрещена: «нет данных» = нет строки.
- **Только 1D-разрезы** — зафиксировано в контракте текстом. 2D-пересечения — осознанно вне v1.
- UTC-only, дневная гранулярность — без переопределений в конфиге.
- Дубликаты ключа `(metric_id, date, dim_key, dim_value)` — ошибка валидации, не «последний
  выигрывает».

Валидатор данных отчитывается человекочитаемо (UX-текст из старого munin-sync-contract
переиспользуется): сколько метрик найдено в данных, какие `metric_id` из parquet неизвестны
схеме, какие метрики схемы без данных, диапазон дат, ошибки типов/дубликатов.

## 6. Фичи MVP

### Вьюхи (UX из прототипа, код новый)

- **Home** — watchlist и health каталога; алерты считаются движком из последних значений против
  порогов схемы. Демо-театр прототипа (LIVE-бейджи, фейковые таймстампы, fake-ссылки) не переносится.
- **Catalog** — grid/list, фасетные фильтры с каскадными счётчиками (паттерн `passesExcept`),
  Cmd+K палитра, секции по level.
- **Card (drawer)** — формула, пороги, бенчмарки, реальный чарт динамики из parquet,
  дрилл-даун (переключатель разреза → серии по dim_value), рендер MD-контекста,
  экспорт карточки в Markdown «как контекст для LLM».
- **Graph** — filter-driven (ADR-015 сохраняется), рёбра строго по `depends_on` из схемы.
- Compare и Saved-отчёты прототипа — post-MVP.

### Редактор-лайт

Формы по полям `catalog.json` (typed-поля, селекты, пороги) + plain-MD редактор с превью
(CodeMirror) для `context/<id>.md`. Сохранение атомарное через zod-валидацию — невалидное
не записывается. Без WYSIWYG, history UI и git-обвязки: история — это git снаружи.

### MCP-сервер («Хугин-лайт»)

5 read-only инструментов: `list_metrics`, `get_metric` (схема + MD-контекст), `query_values`
(id, период, разрез), `get_dependency_graph` (подграф вокруг метрики), `get_alerts`.
Отдельный Node-процесс, который Claude Desktop запускает сам через свой конфиг.
Портфолио-сценарий: «спроси Claude, почему просел GGR».

### Синтетика вместо фейков

`scripts/generate-demo-data.ts`: год дневных данных по контракту — тренд + сезонность + шум
на метрику, значения согласованы через граф зависимостей; 2-3 сценарных инцидента (бонус-абуз
в LatAm, тех-инцидент) каскадом по зависимым метрикам. Это и демо-данные, и smoke-тест графа.
Функции `fakeValue`/`fakeDelta` и весь рантайм-фейк уходят полностью.

## 9. Тесты, ошибки, открытые вопросы

**Тесты:** Vitest на `core` (валидатор схемы, граф, slug-правила, пороги) и `engine`
(контракт-ридер и валидатор данных на parquet-фикстурах, включая битые). UI-тесты — не в MVP.

**Принцип обработки ошибок:** любая проблема vault'а (битый JSON, неизвестные id, кривой
parquet) — человекочитаемый отчёт валидатора в UI, не крэш и не пустой экран. Это и есть
«булетпруф» как фича продукта.

**Открытые вопросы (не блокируют M0-M2):**

1. **Атрибуция исходной Notion-коллекции** — решает владелец до публичного анонса (§7 п.2).
2. **Правила slug для `metric_id`** при миграции сида (транслитерация русских имён vs
   англификация) — решается в плане M1.
3. **Electron vs Tauri** — выбран Electron (один язык со всем стеком, нативный DuckDB через
   официальный `@duckdb/node-api`); Tauri отвергнут осознанно: два языка и IPC-мост замедлят
   соло-разработку. Перепроверять не планируем.
