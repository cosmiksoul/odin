## Что такое ODIN (30 секунд)

**ODIN — privacy-first BI с локальной LLM** для аналитиков в регулируемых индустриях (iGaming, финтех, healthcare).

Технически:
- **Локальная desktop-app** на Electron + React (как Obsidian)
- **Данные пользователя на его машине** — никакого cloud
- **LLM bring-your-own** — Ollama / Claude Desktop / OpenAI-compatible
- **Push-based интеграция с ДВХ** через Parquet-витрину (не pull-коннекторы)
- **Семантический слой** — MD-файлы + локальный git, под капотом
- **In-app WYSIWYG редактор** для аналитиков (без знания git/MD/CLI)

Подробности — `docs/context/architecture.md`. Концепция и метафора (Один + два ворона: Хугин и Мунин) — `docs/context/concept.md`.

## Критичные решения, которые нельзя нарушать

Эти решения зафиксированы в ADR. **Если задача спринта противоречит одному из них — остановись и подними вопрос**, не пытайся реализовать.

### ADR-008: Local-first архитектура

**Никакого cloud-зависимого backend'а в основном пути.** Всё на машине пользователя.

**Что запрещено:**
- ❌ Cloud OAuth для входа в саму ODIN ("Login with Google" — нет)
- ❌ Hosted backend от Anthropic / нашей стороны (нет Vercel/Railway/Fly.io в основном пути)
- ❌ Multi-tenant cloud architecture (нет учёток в облаке)
- ❌ Серверный API key для LLM, proxy endpoints

**Что разрешено (важно — не путай с запретом!):**
- ✅ Локальный credentials store (см. ADR-012) — для S3-ключей, API-токенов, git токенов
- ✅ Push/pull к git remote (см. ADR-013) — пользователь сам подключает свой git
- ✅ Cloud LLM через user-supplied API key (Anthropic, OpenAI) — в credentials store
- ✅ Sync витрины из S3 / HTTP / SFTP — это user-controlled источник данных, не наш backend

### ADR-010: Push-based интеграция через Parquet-витрину

**ODIN не подключается к ДВХ компании напрямую.**

- ❌ Нет коннекторов к Snowflake / BigQuery / ClickHouse / Redshift
- ❌ Нет credential management для prod-систем компании
- ❌ Нет SQL-генерации против чужих БД
- ✅ Компания готовит Parquet в схеме из `munin-sync-contract.md`, ODIN читает

### ADR-011: In-app WYSIWYG редактор — обязательная часть MVP

**Пользователь не работает с git/MD/CLI напрямую.**

- ❌ Нет ссылок "edit on GitHub" / "open in VS Code" из карточки
- ❌ Нет в UX слов "коммит", "ветка", "merge", "push"
- ❌ Не отправляй пользователя править MD-файл снаружи
- ✅ WYSIWYG-редактор внутри ODIN, frontmatter как формочки, body как rich-text
- ✅ Под капотом — MD-файл + локальный git commit (но это не показано в UX)

### ADR-007: Мунин = MD + git + Parquet

**Семантика метрик — MD-файлы. Значения — Parquet через DuckDB.**

- ❌ Нет хранения спек в SQLite / своей БД / JSON
- ❌ Нет своей custom-таблицы версий — git справляется
- ✅ `<vault>/metrics/<metric_id>/spec.md` — спека
- ✅ `<vault>/metrics/<metric_id>/anomaly-*.md` — инциденты
- ✅ `<vault>/data/odin.duckdb` — значения

### ADR-012: Credentials store

**OS-native secret store для всех секретов.**

- ✅ `keytar` library, macOS Keychain / Windows Credential Manager / Linux secret-tool
- ✅ Service name: "ODIN", account names: `aws.s3.<bucket>.access_key`, `llm.anthropic.api_key`, и т.д.
- ❌ Никаких credentials в config.json, .env, или vault-файлах
- ❌ Никакого собственного шифрования / master-password (есть исключения для Linux fallback)

### ADR-013: Shared vault — поэтапная реализация

**В MVP — single-user + опциональный git remote.**

**Что в MVP:**
- ✅ Manual push/pull к remote git через Settings → Vault sync
- ✅ Auto-pull при старте (опция)
- ✅ Auto-push при save (опция, off by default)
- ✅ `created_by` в frontmatter anomaly-заметок (архитектурный hint для будущего RBAC)

**Что НЕ в MVP:**
- ❌ Auto-write Хугина в shared vault (только "Save as anomaly note" по кнопке)
- ❌ Conflict resolution UI внутри ODIN (fallback на git CLI)
- ❌ RBAC (видимость метрик по ролям)
- ❌ Real-time sync (как Obsidian Sync)
- ❌ Multi-user awareness ("Анна редактирует GGR прямо сейчас")

Полный список ADR (13 решений + 1 отозванное) — `docs/context/decisions-log.md`.

## Поведенческие правила

> Правила направлены против типичных ошибок LLM при кодинге: излишняя сложность, незаявленные изменения, додумывание задачи, отсутствие критериев проверки.
>
> **Trade-off:** правила склоняют к осторожности в ущерб скорости. Для тривиальных задач используй здравый смысл.

### 1. Думай до того как писать код

**Не угадывай. Не скрывай неуверенность. Озвучивай trade-off'ы.**

До реализации:

- Заявляй свои предположения явно. Если есть сомнение — спрашивай.
- Если есть несколько интерпретаций задачи — покажи их, не выбирай молча.
- Если есть более простой подход — скажи об этом. Возражай когда есть основания.
- Если что-то непонятно — остановись. Назови что именно непонятно. Спроси.

### 2. Простота прежде всего

**Минимум кода, который решает задачу. Ничего спекулятивного.**

- Никаких фич сверх запрошенного.
- Никаких абстракций для одноразового кода.
- Никакой "гибкости" или "конфигурируемости" если не просили.
- Никакой обработки ошибок для невозможных сценариев.
- Если написал 200 строк, а можно было 50 — перепиши.

Спроси себя: "Сказал бы senior-инженер что это переусложнено?" Если да — упрощай.

### 3. Хирургические изменения

**Трогай только то что нужно. Убирай только свой мусор.**

При редактировании существующего кода:

- Не "улучшай" соседний код, комментарии или форматирование.
- Не рефактори то что не сломано.
- Подстраивайся под существующий стиль, даже если ты бы сделал иначе.
- Если заметил несвязанный мёртвый код — упомяни, не удаляй.

Когда твои изменения создают сирот:

- Удали импорты/переменные/функции, которые **твои** изменения сделали неиспользуемыми.
- Не удаляй pre-existing мёртвый код без явной просьбы.

**Тест:** каждая изменённая строка должна прямо отслеживаться к запросу пользователя.

### 4. Цель-driven выполнение

**Определи критерии успеха. Зацикливайся пока не проверишь.**

Превращай задачи в проверяемые цели:

- "Добавь валидацию" → "Напиши тесты на невалидные входы, потом сделай чтобы они проходили"
- "Исправь баг" → "Напиши тест который воспроизводит баг, потом сделай чтобы он проходил"
- "Отрефактори X" → "Убедись что тесты проходят до и после"

Для многошаговых задач — короткий план:

```
1. [Шаг] → проверка: [как]
2. [Шаг] → проверка: [как]
3. [Шаг] → проверка: [как]
```

Сильные критерии успеха позволяют циклиться независимо. Слабые ("сделай чтобы работало") требуют постоянных уточнений.

## ODIN-specific правила

Помимо универсальных правил выше, есть проектные особенности.

### Прототип Catalog — temporary base, не финальный код

В `src/` лежит код от **Claude Design** (первый прототип v0.1). Это **стартовая точка**, не финальный код.

- При работе со spec-карточками, Graph view, Saved view — этот код можно править свободно
- При значимых архитектурных изменениях — сверяйся с `docs/context/architecture.md`
- Если код прототипа противоречит ADR — следуй ADR, прототип переделывай
- Не сохраняй legacy-решения "потому что было так" — был прототип, теперь продакшн

### Frontmatter MD-файлов — критичная схема

Метрики хранятся как `spec.md` с YAML-frontmatter. Любое изменение схемы frontmatter — это **breaking change** для всех 236 метрик и для WYSIWYG-редактора (формочки привязаны к полям).

- При добавлении нового поля во frontmatter — обсуждай до реализации
- При удалении/переименовании поля — это миграция, не "просто правка"
- Поле `parent_metric_id` и `filter` для derived metrics — см. `docs/project/JTBD-full-scope.md` секцию "Производные метрики"

### Anomaly-заметки — формат тоже стабилизирован + created_by обязательно

Та же логика что для spec.md. Frontmatter anomaly заметок — стабильная схема:

- Обязательные поля: `type: anomaly`, `metric_id`, `date_detected`, `severity`, **`created_by`**
- Опциональные: `date_resolved`, `dimensions`, `root_cause`, `also_visible_in` (для derived metrics)
- **`created_by` обязательно даже в single-user MVP** — это архитектурный hint для будущего RBAC (ADR-013, Этап 3). Берётся из git config или Settings → Profile.

### Credentials — только через credentials store

Любые секреты (S3, API keys, git tokens) хранятся **только** в OS-native secret store через `keytar` (см. ADR-012):

- ❌ Не складывай credentials в config.json, .env, vault-файлы
- ❌ Не делай свою crypto / master-password
- ✅ `keytar.setPassword('ODIN', '<account-name>', '<secret>')`
- ✅ В UI показывай только "Set / Not set", никогда не показывай сам секрет

### Работа с Parquet — только через DuckDB

- Не используй pyarrow / pandas / другие либы для чтения Parquet в основном коде ODIN
- DuckDB embedded через `@duckdb/node-api` — единственный путь
- Python используется **только** для генерации синтетики offline (`data-generation/`)

### Электрон — single process logic

UI и main process — общие. MCP-сервер (Хугин) — child process самого Electron.

- Не запускай MCP как отдельный npx-пакет
- Не разделяй на microservices
- Архитектурное правило: **одна иконка = одна программа**

### Git — обёртка через simple-git, не raw shell

Все операции с git (commit, log, diff, push, pull) — через `simple-git` Node bindings:

- ❌ Не вызывай `child_process.exec('git ...')` напрямую
- ✅ `simpleGit(vaultPath).commit(...)`, `.log(...)`, `.push(...)`
- В UX — никогда не показывай git-терминологию (commit/branch/merge), это под капотом

### Удаление файлов в репо — через `git rm`, не `rm`

Когда в спринте удаляется файл, который был в `git ls-files` (т.е. был в HEAD) — используй `git rm <file>`, **не** просто `rm <file>`. Иначе:

- `git status` показывает странное `modified: <file>` вместо чистого `deleted: <file>`
- При `git add .` (точка) удаление **не попадает** в индекс — пользователю придётся вспомнить про `git add -A`
- Это не баг, но это создаёт лишний шаг и возможность забыть

Это правило действует независимо от того, делаешь ли ты сам commit (обычно нет — это зона пользователя). Просто оставь рабочий tree в состоянии, где `git status` чистый и понятный.

## Workflow спринта (Dev Cycle)

Подробный цикл — `docs/project/Dev-Cycle.md`. Краткая версия для тебя:

1. **Получаешь** `docs/project/sprint-N-prompt.md` — это твоё задание на спринт
2. **Реализуешь** что в prompt'е (не больше, не меньше)
3. **Пишешь отчёт** `docs/project/sprint-report-N.md` по стандартному шаблону
4. **Останавливаешься.** Пользователь и Cowork делают QA, могут вернуть на фикс через `sprint-N-fix-prompt.md`

**Важно:**
- Каждое крупное изменение скоупа в процессе — стоп, задай вопрос
- Не "улучшай" что-то выходящее за prompt
- Если найден архитектурный риск, не отражённый в ADR — стоп, обсуди

## Типичные ошибочные импульсы (anti-patterns)

Эти соблазны возникают часто. Они **все противоречат ADR** или ограничениям MVP:

| Импульс | Почему нет |
|---|---|
| "Давай задеплоим демку на Vercel" | ADR-008: local-first, нет hosted backend |
| "Подключаемся к Snowflake коннектором" | ADR-010: push-based only |
| "Юзер пускай скачает MD и поправит в VS Code" | ADR-011: WYSIWYG inside ODIN |
| "Создам Docker-compose с Postgres" | ADR-008: всё локально, без cloud-инфраструктуры |
| "Отрефакторим заодно вот этот участок" | Правило 3 (Surgical Changes) — не трогай |
| "Добавлю cloud auth (Login with Google)" | ADR-008: нет cloud OAuth для apки |
| "Сделаем наш REST API на Node + Express" | Нет нашего backend. Локальное приложение. |
| "Захардкодим S3 credentials в .env" | ADR-012: credentials в OS-native store, не в файлах |
| "Хугин сам будет коммитить anomaly в shared vault" | ADR-013: в MVP только "Save as note" по кнопке, auto-write — Этап 3 |
| "Сделаем UI разрешения git-конфликтов" | ADR-013: в MVP fallback на git CLI, conflict UI — Этап 2 |
| "Multi-user аккаунты с ролями" | ADR-013: RBAC в Этапе 3 (post-MVP), но `created_by` в frontmatter уже сейчас |
| "Real-time sync как у Obsidian Sync" | ADR-013: только manual push/pull + auto-pull при старте в MVP |

**Если точно нужно** одно из этого — это серьёзное архитектурное изменение. Останавливайся, обсуждай, обновляй ADR **до** реализации.

## Финальная проверка перед коммитом

Перед тем как объявить задачу выполненной:

- [ ] Каждое изменение трассируется к sprint prompt
- [ ] Не добавлено сверх запрошенного
- [ ] Не удалено pre-existing что не относится к задаче
- [ ] Сирот после твоих изменений (мёртвые импорты/переменные) — удалил
- [ ] Тесты для нового поведения — есть и проходят
- [ ] Существующие тесты — проходят
- [ ] Sprint report готов

Если задача с UI:
- [ ] Открыл в Electron, проверил визуально
- [ ] Проверил в светлой и тёмной теме
- [ ] Скриншоты в sprint report (если уместно)

## Признаки что эти правила работают

- Меньше ненужных изменений в diff'ах
- Меньше переписываний из-за переусложнения
- Уточняющие вопросы появляются **до** реализации, а не **после** ошибок
- Не было ни одной попытки добавить cloud-зависимый backend (cloud OAuth, hosted API)
- Credentials идут через keytar, не через файлы
- Каждый коммит чёткий и имеет одну цель
