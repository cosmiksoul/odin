# Data Sample — структура и примеры метрик

Этот файл — компактное приложение к ТЗ для **Claude Design**. Полный JSON (236 метрик) не нужен на этапе дизайна: достаточно понять схему данных и увидеть 3 репрезентативных примера метрик разной природы. Полный JSON пойдёт в Claude Code на следующем этапе.

---

## Схема одной метрики

Каждая метрика — объект с 21 полем. Типы полей в скобках.

| Поле | Тип | Примечание |
|---|---|---|
| `Метрика` | string | Название, уникальный идентификатор |
| `Категория` | select, 31 значение | Revenue, Retention, Bonuses, Sportsbook, Tech / Reliability, … |
| `Уровень (L1/L2/L3)` | select | L1 (58) / L2 (126) / L3 (52) |
| `Приоритет` | select | Must (62) / Should (124) / Nice (50) |
| `Частота` | select | Daily / Weekly / Monthly / Real-time / Per event / Quarterly |
| `Owner` | string, 23 значения | Product, Marketing, CRM, Finance, Tech, … |
| `Источник данных` | select | BI, Analytics, Payments, Betting Engine, Monitoring, … |
| `Важность` | text, 1–3 предл. | Бизнес-контекст, почему метрика важна |
| `Формула / Определение` | text, 1 строка | Пример: `GGR / Total Wager × 100%` |
| `Числитель` | text, 1 строка | Например: `GGR` |
| `Знаменатель` | text, 1 строка или `—` | Например: `Total Wager` |
| `Bench: Tier-1 (EU/AU)` | text | Диапазон, иногда с пояснением |
| `Bench: LatAm/CIS` | text | — |
| `Bench: India/SEA` | text | — |
| `Alert (Red)` | text, короткий | Пример: `<2% casino` или `Drop >20% WoW` |
| `Alert (Yellow)` | text, короткий | — |
| `Ловушки` | text, 3–5 пунктов | Нумерованный список через `\n`, пункты длинные (до 200 симв.) |
| `Как улучшать` | text, 3–5 пунктов | Нумерованный список через `\n` |
| `Зависимости` | text, comma-separated | Имена других метрик. **Критично для графа связей.** Может быть пустым |
| `Разрезы (теги)` | array of strings | Оригинальные теги из Notion — для отображения в карточке |
| `Разрезы (нормализованные)` | array of strings | Канонические теги — для фильтрации. Топ-10: geo, сегмент, канал, продукт, метод платежа, tier, тип бонуса, device, провайдер, игра |

## Пример 2: Метрика Active User Base, L1 Must (с бенчмарками-диапазонами)

```json
{
  "Метрика": "FTD Count",
  "Категория": "Active User Base",
  "Уровень (L1/L2/L3)": "L1",
  "Приоритет": "Must",
  "Частота": "Daily",
  "Owner": "Marketing",
  "Источник данных": "Payments",
  "Важность": "Количество первых депозитов. Главный показатель эффективности связки UA + product. Топ-3 KPI для бизнеса",
  "Формула / Определение": "Количество первых депозитов",
  "Числитель": "Count of first deposits",
  "Знаменатель": "—",
  "Bench: Tier-1 (EU/AU)": "1–5% of visits, CPA ~$30–80",
  "Bench: LatAm/CIS": "1–4% of visits, CPA ~$15–40",
  "Bench: India/SEA": "1–3% of visits, CPA ~$5–20",
  "Alert (Red)": "Drop >20% WoW",
  "Alert (Yellow)": "Drop >10% WoW",
  "Ловушки": "1) FTD count без LTV когорты = метрика тщеславия. 1000 FTD × ~$5 LTV < 200 FTD × ~$100 LTV\n2) FTD всплески от промо-акций могут дать некачественных игроков. Мониторить D7 retention по FTD когорты",
  "Как улучшать": "1) Рост Reg2FTD конверсии (обычно дешевле чем рост регистраций)\n2) Расширение способы оплаты для uncovered сегментов",
  "Зависимости": "",
  "Разрезы (теги)": ["ГЕО", "канал", "платёжка"],
  "Разрезы (нормализованные)": ["geo", "канал", "метод платежа"]
}
```

**Обрати внимание в этом примере:**
- Поле `Зависимости` пустое — не у всех метрик есть связи. Дизайн карточки должен выдерживать отсутствие блока "Связанные метрики"
- Знаменатель `—` означает "не применимо" (метрика count, не ratio). Показывать прочерк или скрывать строку — дизайнерское решение
- Бенчмарки заметно отличаются между регионами (CPA $30–80 vs $5–20) — визуализация разницы может быть сильным ходом

  "Зависимости": "",
  "Разрезы (теги)": ["cashier", "Сервис (platform, games)"],

**Обрати внимание:**
- Все три бенчмарка одинаковые (`>99.9%`) — для universal-метрик вроде uptime регион не имеет значения. Дизайн не должен "кричать" при одинаковых значениях, лучше визуально объединять.
- Разрезы содержат `"Сервис (platform, games)"` с запятой внутри — это валидное значение (а не ошибка парсинга). Если рендерить теги через split по запятой — сломается. Теги уже приходят как array.

## Edge cases для карточки метрики

При дизайне детальной карточки стоит предусмотреть:

1. **Пустые Зависимости** (~3% метрик): блок "Связанные метрики" должен корректно скрываться или показывать plain text.
2. **Одинаковые бенчмарки во всех трёх регионах:** визуальное объединение, чтобы не выглядело "три раза одно и то же".
3. **Длинные Ловушки (до 5 пунктов по 200 символов):** в mobile drawer стоит сделать collapsible-блок.
4. **Composite Owner ("Product / CRM"):** в Ownership map такие owners — отдельные сущности, не двойные. 23 уникальных значения Owner в сумме.
5. **Метрики без числителя/знаменателя** (метрики-count, не ratio): знаменатель приходит как `—`, не как пустая строка.
6. **Метрика с именем, содержащим слэш** (`GGR Margin / Hold %`, `Retention D1 / D7 / D30`): в URL-параметрах нужно URL-encoding.
