# ТЗ: Interactive Data Governance Reference — iGaming (Fantasy v2)

## Контекст и цель

Это вторая итерация fantasy-reference проекта по iGaming-аналитике от того же автора, что и [fantasy-igaming-data-department](https://cosmiksoul.github.io/fantasy-igaming-data-department/). Ключевое отличие от v1: вместо самостоятельно придуманной структуры метрик мы строим **Data Governance-слой поверх существующего open-source справочника метрик** от Андрея Свиридкова (см. раздел Attribution ниже).

**Позиционирование проекта (важно для дизайна):**

Это **НЕ** переупаковка справочника Свиридкова в красивый UI. Это **Data Governance framework**, где справочник Свиридкова выступает **источником данных о метриках**. Основная ценность, которую проект демонстрирует — это:

1. **Governance-слой:** Ownership map, Data Sources Catalog, Metric Maturity methodology, DQ framework — всё это собственная аналитическая работа автора портфолио, не автора источника
2. **Interactive data architecture:** поиск, многомерная фильтрация, граф зависимостей между метриками, shareable URLs — product/UX-вклад автора
3. **Data engineering:** нормализация тегов (с 103 сырых до 39 канонических), фикс артефактов парсинга, структурирование flat-таблицы в многомерный каталог
4. **Cross-regional analytics:** визуализация различий бенчмарков между Tier-1 / LatAm-CIS / India-SEA — аналитический инсайт поверх данных

**Цели интерактивного справочника:**
1. Быстро найти нужную метрику среди 236+ через поиск и фильтры
2. Изучить карточку метрики со всеми аспектами (определение, бенчмарки, ловушки, alert-уровни, зависимости)
3. Увидеть метрику в контексте Data Governance (owner, источник, разрезы, связанные метрики)
4. Понять методологию L1/L2/L3 и приоритизации Must/Should/Nice
5. В будущем — подключить реальные данные и расширить (scope v2)

**Целевая аудитория:** Product/Data analysts, CDO/Head of Data, внешние консультанты в iGaming. Рекрутёры и потенциальные работодатели, просматривающие портфолио автора.

**Deploy target:** GitHub Pages (static hosting, single-page application).

## Ключевой принцип: "fantasy" с честностью

Это фэнтези-справочник. Это значит:
- **Формулы** — берутся из реальной таблицы метрик (поле "Формула / Определение", "Числитель", "Знаменатель"). Где формула абстрактная ("GGR / Total Wager × 100%") — так и оставляем, это норма.
- **SQL-примеры не нужны.** Нет реальной data model, нет реальных таблиц — SQL был бы фейк. Вместо SQL в карточке метрики показываем блок "Формула" как псевдо-код/концепт.
- **Benchmarks — как есть из источника.** Три региональных бенчмарка (Tier-1 EU/AU, LatAm/CIS, India/SEA) — они уже спаршены.
- **Alert thresholds — как есть из источника.** Red и Yellow пороги заполнены для всех 236 метрик.
- **Honest labels.** В футер/about добавить пометку: "Формулы и бенчмарки — reference values, не привязаны к реальной системе. Для подключения реальных данных требуется шаг 2".

## Источник данных

**Файл:** `igaming_metrics_normalized.json` — итоговая версия с нормализованными тегами и восстановленными артефактами парсинга. Расположить в `public/data/` или аналогичной папке.

Структура (реальная, уже есть):

```json
  "collection_title": "Список метрик iGaming",
  "row_count": 236,
  "columns": [{"name": "...", "type": "..."}],
  "rows": [
    {
      "_id": "...",
      "Метрика": "GGR Margin / Hold %",
      "Категория": "Revenue",
      "Уровень (L1/L2/L3)": "L1",
      "Приоритет": "Must",
      "Частота": "Daily",
      "Owner": "Product",
      "Источник данных": "BI",
      "Важность": "Процент удержания, сколько от каждой ставки...",
      "Формула / Определение": "GGR / Total Wager × 100%",
      "Числитель": "GGR",
      "Знаменатель": "Total Wager",
      "Bench: Tier-1 (EU/AU)": "Casino 3–5%, Sports 6–8%",
      "Bench: LatAm/CIS": "Casino 3–5%, Sports 5–7%",
      "Bench: India/SEA": "Casino 3–5%, Sports 5–7%",
      "Alert (Red)": "<2% casino",
      "Alert (Yellow)": "<3% casino",

      "Как улучшать": "1) Смещать набор игр...\n2)...",
      "Зависимости": "GGR, Total Wager (Casino)",
      "Разрезы (теги)": ["ГЕО", "продукт"]
    }
  ]
}
```

**Известные характеристики данных:**
- 236 метрик, 31 категория, 100% заполненность ключевых полей
- Категории: Casino/Games (19), Bonuses (17), Retention (13), Funnel Product (13), Unit Economics (13), Sportsbook (12), Tech/Reliability (11), Risk/Antifraud (10), Revenue (10), CRM (10), Payments Deposits (9), Acquisition UA (8), Active User Base (8), Responsible Gambling (7), VIP (7), Payments Withdrawals (7), Compliance (6), UX Friction (6), Loyalty (5), Affiliate (5), Data Quality (5), Support (5), Experimentation (5), и ещё ~8 небольших
- Уровни: L1 (58), L2 (126), L3 (52)
- Приоритеты: Must (62), Should (124), Nice (50)
- Owners (топ): Product (84), Marketing (21), CRM (16), Analytics (13), Compliance (13), Risk (12), Payments (11), Finance (10), Tech (8), VIP (8)
- Источники данных (топ): BI (79), Analytics (29), Payments (19), Betting Engine (11), Monitoring (11), Game Aggregator (11), Bonus Engine (11)
- Частоты: Monthly (92), Weekly (75), Daily (62), Per event (4), Real-time (2), Quarterly (1)

**Теги-разрезы — нормализованы.** В исходных "Разрезы (теги)" было 103 уникальных тега со смешением ru/en/регистров (ГЕО / гео / geo, продукт / product / Provider / Провайдер). Мы применили словарь нормализации и получили 39 канонических тегов. Оба поля теперь есть в JSON:
- `Разрезы (теги)` — оригинальные теги, как в источнике (для честности в карточке метрики)
- `Разрезы (нормализованные)` — канонические теги для фильтрации

**Топ-10 нормализованных тегов (для фильтра в sidebar):**
- `geo` (139) — объединяет ГЕО/гео/geo/юрисдикция/рынок
- `сегмент` (69) — сегмент/segment/когорта/cohort/new vs returning
- `канал` (50) — канал/channel
- `продукт` (37) — продукт/product/last product
- `метод платежа` (21) — метод/method/Платёжный метод/платёжка/Метод вывода/cashier
- `tier` (20) — tier/VIP tier/Tier transition
- `тип бонуса` (19) — Тип бонуса/Bonus type/тип оффера/Тип промо/redemption type
- `device` (16) — device/OS
- `провайдер` (16) — Провайдер/provider
- `игра` (15) — игра/Game/game type/тип игры/category

**Правила для UI:**
- **Фильтр в sidebar** использует только поле `Разрезы (нормализованные)` и только топ-10 значений
- **В карточке метрики** показываем оригинальные теги из `Разрезы (теги)` (чтобы сохранить нюансы — "VIP tier" точнее, чем просто "tier")
- **Metadata в JSON** уже содержит готовые массивы `normalized_tags_top10` и `normalized_tags_all` — Claude Code может их использовать напрямую

**Артефакт парсинга зафиксен.** В исходной базе был тег `"Сервис (platform, games)"` с запятой внутри, из-за которого парсер разбивал его на два: `"Сервис (platform"` и `"games)"`. В финальном JSON этот артефакт восстановлен в корректный тег `"Сервис (platform, games)"`.

**Одна метрика ("Метрика") = одна карточка.** Это primary key.

## Архитектура приложения (страницы)

Single-page app с hash-роутингом. Четыре экрана:

1. **Home** (`#/`) — hero, юзеркейсы, методология, региональные бенчмарки, распределения
2. **Metrics Catalog** (`#/metrics`) — поиск и фильтрация по 236 метрикам с drawer-карточкой
3. **Data Governance Framework** (`#/governance`) — ownership, sources, maturity levels, DQ framework
4. **About** (`#/about`) — атрибуция источника, кредиты методологии, технический стек

Topbar navigation: Home · Catalog · Governance · About. GitHub-ссылка — иконкой справа.

Точка входа. Не длинная портянка — обзорная страница с визуализациями распределений и быстрой навигацией. Также несёт образовательную функцию: знакомит с методологией, прежде чем пользователь уйдёт в каталог.

**Блоки (в порядке сверху вниз):**

**C. "Как использовать этот справочник" — 4 юзеркейса (адаптированы под интерактивный формат)**

Каждый юзеркейс — карточка с короткой пошаговой инструкцией, заканчивающаяся CTA-кнопкой с preset-фильтрами:

2. **Диагностика проблемы**
   "Нашёл просадку в L1-метрике? Открой её карточку, перейди по зависимостям к связанным L2-метрикам, используй разрезы (гео/продукт/канал) для сегментации. Блок 'Ловушки' поможет исключить ошибку интерпретации."

   → CTA: "Открыть каталог метрик"

4. **Онбординг нового аналитика / PM**

   → CTA: "Фильтр по Owner-командам"

**D. Методология — два парных блока**

Это ключевой образовательный блок. Рендер — две колонки side-by-side.

*Блок D1: Система уровней (L1/L2/L3)*
- Визуализация: пирамида из 3 уровней (L1 сверху, L3 снизу) с количеством метрик в каждом (58 / 126 / 52)
- Подпись под пирамидой: "L1 показывает **что** происходит → L2 помогает понять **почему** → L3 указывает **где именно**"
- Описания уровней (перефразированные своими словами, не копия источника):
  - **L1 (58) — Стратегические:** C-level дашборды, ежедневный ревью. Просадка L1 = проблема бизнеса.
  - **L2 (126) — Операционные:** Для продукт/CRM/маркетинг команд. Объясняют динамику L1.
  - **L3 (52) — Диагностические:** Глубокая детализация при расследовании аномалий L2.

*Блок D2: Система приоритизации (Must/Should/Nice)*
- Визуализация: три горизонтальных полосы со счётчиками (62 / 124 / 50)
- Описания:
  - **Must (62) — Внедрять сразу:** Минимум для управления бизнесом. Без них летишь вслепую.
  - **Should (124) — Следующий уровень:** Операционная глубина, контроль качества решений.
  - **Nice (50) — Для зрелых команд:** Advanced analytics, experimentation, deep diagnostics.

**E. Региональные бенчмарки — объяснение**

Блок с 3 карточками (по одной на регион), объясняющий **что означают** три столбца бенчмарков в карточках метрик. Это критично: без этого пользователь видит три столбца и не понимает разницы.

- **Tier-1 (EU / UK / AU):** Зрелые регулируемые рынки. Высокий ARPU, высокие costs (compliance, UA). Benchmarks здесь — "well-established" диапазоны.
- **LatAm / CIS (BR / MX / RU / UA / KZ):** Растущие рынки. Средний ARPU, конкурентная среда, динамика быстрее.
- **India / SEA (IN / BD / PH / ID):** Высокий объём, низкий ARPU. Специфика платёжных методов и регулирования.

⚠ Критический disclaimer под блоком (крупно и явно):
> "Benchmarks — это ориентиры, а не таргеты. Реальные значения зависят от стадии продукта, бизнес-модели, лицензирования, ГЕО-микса. Отклонения в разы — норма."

### 2. Metrics Catalog (`#/metrics`)
Сердце проекта. Поиск и фильтрация по 236 метрикам.

**Левая панель фильтров (sticky sidebar, collapsible на mobile):**
- Search input (fuzzy по Метрике + Важности)
- Чекбоксы по Категории (с count в скобках) — expandable sections
- Чекбоксы по Уровню: L1 / L2 / L3
- Чекбоксы по Приоритету: Must / Should / Nice
- Dropdown по Owner (multi-select)
- Dropdown по Источнику данных (multi-select)
- Чекбоксы по Частоте: Daily / Weekly / Monthly / Real-time / Per event / Quarterly
- Чекбоксы по топ-10 тегам (ГЕО, сегмент, канал, продукт, device, Провайдер, tier, кампания, Тип бонуса, метод)
- Кнопка "Clear all filters"
- Счётчик "Показано X из 236"

**Основная область:**
- Переключатель вида: Grid (карточки) / List (компактная таблица)
- Сортировка: по алфавиту / по уровню / по приоритету / по категории
- **Grid view:** карточки метрик 3 в ряд на desktop, 2 на tablet, 1 на mobile. На карточке: название, категория-бадж, уровень-бадж, приоритет-бадж, краткая "Важность" (первые 120 символов), owner. Клик → открывает детальную карточку.
- **List view:** компактная таблица — Метрика, Категория, L, P, Owner, Частота. Клик по строке → детальная карточка.

**Детальная карточка — открывается как right drawer / side panel (60% ширины экрана, затемнение фона), НЕ отдельная страница.** Причина: возвращаться к отфильтрованному списку через back — плохой UX.

### 3. Data Governance Framework (`#/governance`)
Здесь мы опираемся на лучшую часть предыдущего fantasy-проекта автора (структуру DG-раздела) и связываем её с реальными данными из каталога. Плюс — явно выделяем собственную методологию автора (ownership map, maturity levels, DQ framework) как главную ценность раздела.

**Разделы:**

**A. Ownership Map**
- Таблица: Owner-команда × количество метрик × категории, в которых они работают
- Для каждого owner — "View metrics" → переход на каталог с фильтром
- Сгенерить из данных автоматически: 23 уникальных owner в таблице → агрегировать
- Добавить визуализацию: heatmap Owner × Категория (цвет = количество метрик)
- **Инсайт-блок:** "Top-3 owners (Product 84, Marketing 21, CRM 16) владеют 51% всех метрик — типичная картина в iGaming"

**B. Data Sources Catalog**
- Список источников данных (BI, Analytics, Payments, Betting Engine, Bonus Engine, и т.д.) с описанием и количеством метрик, которые из них берутся
- Описания источников — fantasy/концептуальные, но логичные. Примеры:
  - **Betting Engine** — real-time система обработки ставок. Источник событий `placed_bet`, `settled_bet`, `cashout`. 11 метрик.
  - **Bonus Engine** — управление бонусами, wagering, promo campaigns. 11 метрик.
  - **BI** — централизованный аналитический warehouse, основной источник для отчётности. 79 метрик.
- Клик по источнику → фильтр каталога по нему
- **Disclaimer:** источники — fantasy-концепты для иллюстрации архитектуры; в v2 планируется привязка к реальной data model

**C. Metric Maturity Levels (L1/L2/L3)**

Методологический раздел. Расширенное объяснение иерархии уровней (перефразированное, не копия источника):

*Концепция:* Три уровня иерархии — не по важности, а по **функциональной роли**. L1 отвечает "что", L2 — "почему", L3 — "где".

*Визуализация:* пирамида или ступенчатая диаграмма с примерами метрик из каждого уровня:
- L1: GGR, NGR, MAU, FTD Count, LTV:CAC — метрики, которые видит CEO
- L2: CAC по каналам, Retention D7/D30, Bonus Cost Ratio — метрики, которые объясняют динамику L1
- L3: Sportsbook hold% by market type, Casino RTP variance, KYC completion by step — метрики для глубокого расследования

*Связь с ревью-процессом:*
- L1 → ежедневный CEO dashboard, weekly business review
- L2 → weekly team reviews (product, marketing, CRM)
- L3 → ad-hoc расследования, post-mortems, аномалии

**D. Priority Framework (Must/Should/Nice)**

*Концепция:* Приоритет метрики — это **инструкция внедрения**, не оценка важности. Даже Nice-метрика может быть критична в конкретном контексте; но если её нет, бизнес не развалится.

- **Must (62 метрики) — Day 0.** Без этих метрик ты не можешь принимать операционные решения. Внедрять до запуска продукта или в первую волну миграции на новую BI.
- **Should (124 метрики) — Day 30-90.** После стабилизации Must — добавляют операционную глубину: сегментацию, контроль качества, ранние сигналы.
- **Nice (50 метрик) — Day 180+.** Для продвинутой аналитической зрелости: experimentation framework, advanced retention, guardrail systems.

*Визуализация:* 3 карточки с счётчиками, описанием, примером метрики из каждой группы. Интерактив: клик по группе → фильтр каталога.

**E. Data Quality Framework**
- 5 измерений: Completeness, Accuracy, Timeliness, Consistency, Uniqueness
- Концептуальные описания с примерами из текущих метрик
- Пример: "Для метрики **GGR Margin** критична Consistency — hold% должен сходиться в BI и в Betting Engine. Расхождение > 0.5% — сигнал на расследование (см. ловушки в карточке метрики)."
- Пример: "Для метрики **FTD Count** критична Uniqueness — один игрок = один FTD. Duplicates раздувают CAC-метрики и искажают attribution."

**F. Update Cadence**
- Визуализация: Daily (62) / Weekly (75) / Monthly (92) / Real-time (2) / Per event (4) / Quarterly (1)
- Инсайт: "Большинство метрик iGaming (71%) считаются weekly/monthly — высокочастотные обновления нужны только для operational-слоя (deposits, active users, incident metrics)"

**НЕ делаем в v1 (явно исключить):**
- RBAC / Access Policies — в старом проекте было выдумано, давай честно скажем "scope v2"
- Retention / Data Lifecycle — тоже v2
- Pipeline diagrams — v2

Вместо этого добавить блок "Roadmap v2":
- Привязка к реальной data model (dbt models, ClickHouse tables)
- SQL-примеры на основе реальной модели
- Access control и PII handling
- Pipeline и retention policies
- Интеграция с реальным DataHub / метрик-каталогом

## Design system

### Стилистика
- **Темная тема по умолчанию** (старый проект — как раз в тёмной работает хорошо)
- **Acceнт — не скучный корпоративный синий.** Предлагаю палитру с основным цветом `#10b981` (emerald) или `#06b6d4` (cyan) — для iGaming domain отсылает к азарту, но не кричащий. Фон `#0a0a0a`, карточки `#111111`, borders `#1f1f1f`.
- **Типографика:** заголовки — Inter или Geist Sans, моноширинный (формулы, коды) — JetBrains Mono или Geist Mono
- **Бейджи-категории цветные** — каждая из 31 категории имеет уникальный hue (генерируется программно от hash названия). Но приоритет/уровень — фиксированные цвета: Must=red-ish, Should=amber, Nice=gray; L1=emerald, L2=blue, L3=purple.

### UI components
- Нужны: карточки, бейджи, drawer/sheet, search input с иконкой, multi-select dropdown, checkbox list, tabs, tooltips, copy-to-clipboard на ссылки
- **Не использовать heavy UI-библиотеку целиком.** Или shadcn/ui (копированием компонентов), или кастомные с Tailwind. Избегать Material UI — слишком "энтерпрайз".

### Анимации
- Умеренные: drawer slide-in 200ms, card hover lift, filter count animate
- **НЕ делать** fancy parallax / scroll-triggered effects — для reference работы это мешает

## Техническая реализация

### Stack
- **React + Vite + TypeScript** (типизация под структуру метрики даст compile-time защиту)
- **Tailwind CSS**
- **Wouter или React Router** для hash-routing (GitHub Pages friendly)
- **Zustand или URL params** для состояния фильтров (Zustand + синхронизация с URL)

- **Recharts или Chart.js** для визуализаций на Home (treemap — recharts@latest)
- **No backend** — чисто статика, JSON подгружается при старте

- 236 карточек одновременно — норм без виртуализации
- Но если List view — табличка на 236 строк, можно оставить без виртуализации, или использовать TanStack Virtual если заметим проблемы
- JSON ~500KB — подгрузить разом, не чанковать
- Лениво грузить чарты на Home (динамический import recharts)

### SEO / sharing
- Мета-теги для соцсетей (OG image — статичный preview)
- Если открыта конкретная метрика — title страницы меняется на "Metric name — iGaming Reference"

## Что является success criteria

Проект считается успешным, если:
1. **Поиск "GGR" находит все 15+ связанных метрик за <100ms** и показывает их карточки
2. **Фильтр "Owner = CRM + Уровень = L1"** показывает корректный subset за <50ms
3. **Клик на зависимость в карточке** открывает карточку зависимой метрики (работает граф)
4. **Shareable URL** — можно поделиться ссылкой на отфильтрованный список или конкретную метрику
5. **Mobile-friendly** — работает на телефоне (hamburger для фильтров, drawer на всю ширину)
6. **Deploy на GitHub Pages работает** из коробки (vite config с правильным base)
